## 一、概述

OpenNexum 是一个运行在 OpenClaw 上的多 agent 编排框架，专为**单人指挥、多 agent 并行执行**的工作模式设计。

它的核心理念是：**合约驱动执行，知识持续沉淀**。每项工作在启动前先确立可验证的完成标准（Contract），完成后自动将经验写回系统供后续复用。随着时间推移，系统变得越来越了解你的项目和工作习惯。

### 设计背景

OpenNexum 在早期 agent 编排实践的基础上，吸收了 OpenAI 和 Anthropic 在智能体工程上的核心经验：

- **Repo 是唯一的 source of truth**：agent 看不到的信息等于不存在，所有知识必须结构化存储在项目里
- **Generator 与 Evaluator 分离**：做的人不应该评判自己的输出，独立 evaluator 是质量的真正保障
- **harness 组件应当可裁剪**：每个组件都是对模型能力不足的补偿假设，随模型进化可按需删减

## 二、定位与边界

### 2.1 目标用户

**OpenNexum 专为个人开发者设计**，不是通用 multi-agent 平台。

适用场景：
- 个人独立工作，无需多人协作模式
- 项目代码量 ≤ 20 万行，单次开发周期 ≤ 1 个月
- 使用 OpenClaw 作为主控层，调度 Claude Code / Codex / Google CLI 等 CLI agent
- 本机单机运行（Mac Mini），使用 tmux 管理 agent 进程

### 2.2 任务类型

OpenNexum 支持三类工作，共享同一套编排机制：

| 类型 | 示例 |
|------|------|
| **开发类** | 功能开发、Bug 修复、重构、测试、部署 |
| **日常任务类** | 数据整理、研究报告、信息汇总、文档翻译 |
| **创作类** | 写作、内容生成、脚本创作、设计描述 |

### 2.3 非目标

- 不支持多人团队协作
- 不支持分布式多机部署
- 不追求"通用万能"，有明确的用户边界

## 三、核心设计

### 3.1 Contract-Driven 执行

所有工作在启动前必须建立 **Task Contract**：在任何内容被生成之前，先对齐"什么叫完成"。

Contract 定义：
- **Scope**：要做什么、改哪些文件
- **Deliverables**：具体交付物清单
- **Eval Strategy**：用什么方式验证
- **Criteria**：逐条可验证的通过标准

Contract 是 generator 和 evaluator 共同可见的完成定义，不是 orchestrator 单方面写的 prompt。

### 3.2 Generator ≠ Evaluator（GAN 原则）

执行 agent（generator）和评估 agent（evaluator）严格分离：

- generator 专注完成任务
- evaluator 独立验证，不参与执行
- evaluator 被显式调校为"怀疑主义"：任务是找问题，而非确认没问题
- fail 时必须给出具体文件路径、函数名、期望行为 vs 实际行为，不允许模糊结论

### 3.3 知识持续沉淀

**对 agent 来说，context 里看不到的信息等于不存在。**

OpenNexum 通过两层结构确保知识可见、可演化：

**项目层**：每次任务完成后，agent 自动将遇到的问题和发现写入 `docs/lessons/`；harvest 流水线将 lessons 整理回 `AGENTS.md`，让下一个任务的 agent 能看到。

**Skill 层（OpenClaw）**：跨项目的通用规范和反模式库存储在 skill 内部；项目初始化时自动注入到项目结构中；gardener agent 定期将多个项目的经验蒸馏回 skill 层。

### 3.4 渐进式披露（Progressive Disclosure）

AGENTS.md 是目录，不是说明书，保持约 100 行。

> "给 agent 一张地图，而不是一本 1000 页的手册"

详细内容通过 AGENTS.md 的索引按需加载。过多的指导在上下文中会失去效果，甚至造成干扰。

### 3.5 机械约束优于 Prompt 约束

对于不变的规则（提交格式、文件命名、架构层级），用 linter/CI 机械强制执行，error message 里直接写修复指令。规则一旦编码，立即对所有 agent 生效，无需反复在 prompt 中叮嘱。

## 四、系统架构

### 4.1 分层结构

```
┌───────────────────────────────────────────────────┐
│  人类（Telegram / 终端）                           │
│  提需求 · 审核 Contract · 处理升级问题             │
└──────────────────┬────────────────────────────────┘
                   │
┌──────────────────▼────────────────────────────────┐
│  OpenClaw — Orchestrator（编排层）                 │
│                                                    │
│  • 接收需求，制定 Task Contract                    │
│  • 决定 eval strategy，dispatch generator          │
│  • 处理 evaluator 结果（pass/fail 循环）           │
│  • 管理知识流：lesson 写入 → harvest → 同步        │
│  • Telegram 通知人类关键节点                        │
└──┬────┬────┬────┬────┬────┬──────────────────────┘
   │    │    │    │    │    │
┌──▼─┐┌─▼──┐┌▼───┐┌▼───┐┌──▼──┐┌──▼──────┐
│plan││cod ││cod ││cc- ││eval ││gardener │
│    ││ex-1││ex- ││fr- ││     ││         │
│规划││后端││前端││前端││评估 ││知识维护 │
│    ││    ││内部││外部││     ││         │
└────┘└────┘└────┘└────┘└─────┘└─────────┘
```

### 4.2 Agent 角色

| Agent | 默认模型 | 职责 |
|-------|---------|------|
| `plan` | claude-opus | 规划、制定/审核 Contract、拆解任务 |
| `codex-1` | gpt-5.4 | 后端、算法、脚本、部署 |
| `codex-frontend` | gpt-5.4 | 内部工具、admin 类前端（对样式无高要求） |
| `cc-frontend` | claude-sonnet | 外部用户可见的前端 UI（质量优先） |
| `cc-writer` | claude-sonnet | 写作、创作、文档类任务 |
| `eval` | gpt-5.4 | 独立评估，不参与执行；复杂/主观任务可配置为 claude-sonnet |
| `gardener` | claude-sonnet | 定期 harvest lessons，维护知识库 |

**模型路由默认建议（可通过配置覆盖）：**

| 任务类型 | 默认 Agent | 理由 |
|----------|-----------|------|
| 后端 / 算法 / 脚本 | `codex-1` | token 经济 |
| 内部 admin / 后台 UI | `codex-frontend` | 样式要求低，节省 token |
| 外部用户可见 UI | `cc-frontend` | 质量优先 |
| 写作 / 创作 | `cc-writer` | 语言能力 |
| eval（客观验证） | `eval`（gpt-5.4） | 速度 + 经济 |
| eval（主观/创作） | `eval`（claude-sonnet） | 批判性思维 |
| 规划 / Contract 制定 | `plan`（claude-opus） | 全局理解能力 |

> ⚠️ 以上为**建议默认值**。所有 agent 的模型和 CLI 工具均可在项目配置中独立覆盖，系统不强制绑定任何模型。

### 4.3 执行容器

使用 **tmux** 作为 agent 进程容器：
- 轻量，单机无额外依赖
- 随时 attach 查看 agent 实时状态
- 支持多 session 并行
- 通过 dispatch 脚本统一管理生命周期

### 4.4 AGENTS.md 与 CLAUDE.md 同步

项目根目录同时维护两个文件以兼容不同 CLI：

- `AGENTS.md`：Codex CLI 自动加载
- `CLAUDE.md`：Claude Code 自动加载

**`AGENTS.md` 是唯一 source of truth**，`CLAUDE.md` 通过 git post-commit hook 自动同步（`cp AGENTS.md CLAUDE.md`），无需手动维护。

> 规则：永远不要直接编辑 `CLAUDE.md`，改了也会在下次 commit 时被覆盖。

## 五、Task Contract

### 5.1 Contract 格式（YAML）

```yaml
# docs/nexum/contracts/TASK-001.yaml
id: TASK-001
name: "实现用户登录页面"
type: coding          # coding | writing | analysis | creative
created_at: "2026-03-27T09:00:00Z"

scope:
  files:
    - src/pages/login.tsx
    - src/api/auth.ts
  boundaries:
    - "不修改 src/components/ui/ 下的基础组件"
    - "不修改数据库 schema"

deliverables:
  - "登录表单，支持邮箱 + 密码"
  - "调用 POST /api/auth/login"
  - "错误状态展示"
  - "登录成功跳转 /dashboard"

eval_strategy:
  type: e2e             # unit | integration | e2e | review | composite
  tools: [playwright]
  local_url: "http://localhost:3000"   # e2e 任务必填
  criteria:
    - id: C1
      desc: "表单提交后正确调用 /api/auth/login"
      method: "playwright: intercept network request"
      threshold: pass
    - id: C2
      desc: "错误密码时展示错误提示"
      method: "playwright: check error element visible"
      threshold: pass
    - id: C3
      desc: "登录成功后 URL 变为 /dashboard"
      method: "playwright: check URL after submit"
      threshold: pass

generator: codex-frontend
evaluator: eval            # eval 默认用 gpt-5.4；可覆盖为 claude-sonnet
max_iterations: 3
depends_on: []
```

### 5.2 Eval Strategy 选择矩阵

| 任务特征 | 推荐策略 | 默认 Evaluator 模型 |
|----------|----------|-------------------|
| 纯算法/逻辑，有明确输入输出 | `unit` | gpt-5.4 |
| 涉及数据库/API 集成 | `integration` | gpt-5.4 |
| 有前端 UI，用户可交互 | `e2e`（Playwright，需 `local_url`） | gpt-5.4 |
| 多维度复杂功能 | `composite`（组合） | gpt-5.4 |
| 架构/代码质量/安全性 | `review` | gpt-5.4 |
| 写作/创作类 | `review`（标准化评分） | claude-sonnet |
| 日常任务/研究报告 | `review`（完成度核查） | gpt-5.4 |

> `review` 是 eval strategy 的一种类型（`type: review`），不是独立流程。所有验证都通过 evaluator agent 执行，区别只在 criteria 的验证方式。

### 5.3 Contract 制定流程

```
orchestrator 起草 contract draft
    ↓
（如需代码探索）plan agent 审核并补充技术细节
    ↓
orchestrator 最终确认，写入 contracts/TASK-ID.yaml
    ↓
dispatch generator
    ↓
generator 完成后，evaluator 拿 contract 逐条验证
    ↓
全部 pass → mark done
有 fail → feedback 回 generator，进入下一轮
超过 max_iterations → escalate to human
```

### 5.4 Evaluator 调校要求

evaluator prompt 必须包含：
1. **怀疑主义指令**："你的工作是找问题，不是夸奖输出"
2. **逐条验证**：对每条 criterion 必须给出 pass/fail + 具体证据，不允许整体判断
3. **可操作的 fail 反馈**：包含文件路径、函数名、期望行为 vs 实际行为
4. **禁止模糊通过**："整体看起来没问题"不构成通过标准

## 六、知识管理

### 6.1 Lesson 格式

agent 在完成任务过程中，遇到非预见问题或值得记录的模式时，写入：

```markdown
<!-- docs/lessons/YYYY-MM-DD-TASK-ID.md -->
## Lesson: [简短标题]
- trigger: [什么情况触发了这个问题]
- wrong: [错误做法或错误假设]
- right: [正确做法]
- affected: [具体文件/函数/API]        ← 有此字段 = 技术层 lesson
- agent_note: [orchestrator 下次注意]  ← 有此字段 = 流程层 lesson
- tags: [pg, auth, typescript, ...]
```

`affected` 和 `agent_note` 互斥：
- `affected` 有代码路径 → **技术层**，最终进入 AGENTS.md
- `agent_note` 描述 orchestrator 行为 → **流程层**，最终进入 skill 内置知识

### 6.2 Harvest 流水线

每次批次完成后自动触发：

```
on-complete（全部任务 done）→ 触发 harvest
    ├── 读取新增 lesson 文件
    ├── 与现有内容去重（关键词 + 标签匹配）
    ├── 互斥检查（是否与已有规范冲突）
    └── 根据结果分流：
        ├── 纯新增 + 无冲突 → 自动合并 + Telegram 通知"已更新"
        └── 有冲突 or 有歧义 → 生成 diff + Telegram 通知，等人工确认
```

### 6.3 知识层级

| 层级 | 存储位置 | 内容 | 更新方式 |
|------|----------|------|----------|
| 技术层 | `project/docs/lessons/` → `AGENTS.md` | 代码规范、反模式、踩坑 | harvest 批次完成后自动触发 |
| 流程层 | `skill/references/lessons/` | orchestrator 行为优化 | harvest 更新 skill |

### 6.4 Gardener Agent

每批次完成后**自动触发**（也可手动触发）：
- 扫描所有 lesson 文件，识别过时/失效条目
- 合并重复/相似内容
- 更新 AGENTS.md 技术层规范
- 输出变更摘要，人工审核后提交

## 七、项目初始化

### 7.1 命令

```bash
nexum init --project <name> --tech "<stack>" --type <coding|writing|mixed>
```

### 7.2 执行内容

1. 创建标准目录结构
2. 从 skill 模板生成 `AGENTS.md`（≈100 行目录 + 通用工程规范）
3. 自动生成 `CLAUDE.md`（通过 hook 与 AGENTS.md 保持同步，无需手动维护）
4. 从 skill 反模式库提取与 tech stack 匹配的 lessons，注入 AGENTS.md
5. 初始化 `docs/` 目录（design / lessons / nexum/contracts）
6. 初始化 `nexum/active-tasks.json`
7. 安装 git hooks（含 CLAUDE.md 自动同步 + post-commit 信号）

### 7.3 标准项目结构

```
project/
├── AGENTS.md              ← source of truth，≈100 行目录
├── CLAUDE.md              ← 由 hook 自动同步，勿手动编辑
├── ARCHITECTURE.md        ← 架构概览（首次由 plan agent 生成）
├── docs/
│   ├── design/            ← 技术设计文档
│   ├── lessons/           ← 踩坑记录（agent 自动写入）
│   └── nexum/
│       └── contracts/     ← Task Contract 存档（YAML）
└── nexum/
    ├── active-tasks.json  ← 当前批次任务状态
    ├── agent-pool.json    ← agent 状态
    ├── config.json        ← 项目级配置（模型覆盖等）
    └── history/           ← 已归档批次
```

### 7.4 AGENTS.md 内置学习协议

生成的 AGENTS.md 必须包含：

```markdown
## 错误学习协议

遇到非预见错误或值得记录的模式时：

1. 修复或完成任务
2. 写入 docs/lessons/YYYY-MM-DD-<TASK-ID>.md（使用标准模板）
3. 不要直接修改 AGENTS.md — 由 gardener 统一维护

这是强制要求，不是可选项。
```

## 八、任务生命周期

### 8.1 状态机

```
pending ──→ running ──→ evaluating ──→ done
              │               │
              ↓               ↓
            failed ←── eval_fail（未超次数）
              │
              ↓（超过 max_iterations）
           escalated → 通知人类
```

blocked：depends_on 未满足时的等待态，dependencies 全部 done 后自动转 pending。

### 8.2 完整执行循环

```
1. orchestrator 制定 Task Contract
2. 注册任务（active-tasks.json）
3. dispatch generator
4. generator 执行，git commit
5. post-commit hook → push + 写信号
6. on-complete：
   a. 更新任务状态
   b. dispatch evaluator（根据 eval_strategy）
   c. 唤醒 orchestrator
7. orchestrator 处理 eval 结果：
   - pass → mark done → 解锁下游 → dispatch 下一个
   - fail → feedback 回 generator → 进入下一轮
   - 超过 max_iterations → escalate to human
8. 所有任务 done → 汇报 → 自动触发 harvest + gardener
```

### 8.3 并行安全规则

file scope 不重叠的任务可并行 dispatch。dispatch 前检查：
- 新任务 scope 与所有 running 任务 scope 无重叠 → 可并行
- 有重叠 → 等待，或通过 `depends_on` 串行化

## 九、配置系统

### 9.1 项目级配置（nexum/config.json）

每个项目可覆盖默认的模型和 agent 设置：

```json
{
  "agents": {
    "plan": { "cli": "claude", "model": "claude-opus-4-6" },
    "codex-1": { "cli": "codex", "model": "gpt-5.4", "reasoning": "high" },
    "codex-frontend": { "cli": "codex", "model": "gpt-5.4", "reasoning": "medium" },
    "cc-frontend": { "cli": "claude", "model": "claude-sonnet-4-6" },
    "eval": { "cli": "codex", "model": "gpt-5.4", "reasoning": "high" },
    "gardener": { "cli": "claude", "model": "claude-sonnet-4-6" }
  },
  "notify": {
    "target": "8449051145",
    "verbose_dispatch": true
  },
  "harvest": {
    "auto_trigger": true
  }
}
```

> 所有字段均可覆盖，未配置的字段使用 skill 级默认值。

### 9.2 Dispatch 卡片

每次 dispatch 发送：

```
🚀 Dispatched [TASK-ID] → [SESSION]
┣ 📋 Contract: docs/nexum/contracts/TASK-ID.yaml
┣ ⏰ Started:  HH:MM:SS
┣ 🤖 Model:   [模型名]（[reasoning level]）
┣ 📝 Task:    [一行描述]
┗ 🧪 Eval:    [eval_strategy type]
```

## 十、通知分级

| 级别 | 触发条件 | 方式 |
|------|----------|------|
| 🔴 立即 | 超过 max_iterations；需人工决策；涉及 secrets/env | Telegram 立即推送 |
| 🟡 批次 | 全部任务完成；harvest 冲突待审核 | Telegram |
| 🟢 静默 | 常规重试；agent 内部问答；routine fix | 只记录日志 |

## 十一、开发路线图

### Phase 1 — 合约基础（MVP）
- Task Contract YAML 规范与解析
- `nexum init` 命令（项目结构 + AGENTS.md 生成 + hook 安装）
- Contract-based dispatch（含 AGENTS.md/CLAUDE.md 自动同步）
- Evaluator agent（review 模式，Codex 驱动）
- Lesson 写入协议（模板 + AGENTS.md 内置学习指令）
- 项目级 config.json 覆盖机制

### Phase 2 — 验证能力
- Playwright e2e eval（需 local_url）
- Unit / integration test eval（运行 test suite，解析结果）
- composite eval strategy

### Phase 3 — 知识系统
- Harvest 流水线（去重 + 互斥检查 + 自动合并）
- 技术层 vs 流程层自动分流
- Gardener agent（批次完成后自动触发）

### Phase 4 — 任务类型扩展
- Writing / Creative Contract 规范
- 写作类评分标准（design quality / originality / craft）
- Analysis 任务支持

### Phase 5 — 工程加固
- linter / CI 架构约束（自定义规则 + error message 含修复指令）
- Harvest 语义去重（embedding-based）
- 可靠性与性能优化
