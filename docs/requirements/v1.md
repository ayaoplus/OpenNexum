## 一、概述

OpenNexum 是一个运行在 OpenClaw 上的多 agent 编排框架，专为**单人指挥、多 agent 并行执行**的工作模式设计。

它的核心理念是：**合约驱动执行，知识持续沉淀**。每项工作在启动前先确立可验证的完成标准（Contract），完成后自动将经验写回系统供后续复用。随着时间推移，系统变得越来越了解你的项目和工作习惯。

### 设计背景

OpenNexum 在早期 agent 编排实践的基础上，吸收了 OpenAI 和 Anthropic 在智能体工程上的核心经验：

- **Repo 是唯一的 source of truth**：agent 看不到的信息等于不存在，所有知识必须结构化存储在项目里
- **Generator 与 Evaluator 分离**：做的人不应该评判自己的输出，独立 evaluator 是质量的真正保障
- **harness 组件应当可裁剪**：每个组件都是对模型能力不足的补偿假设，随模型进化可按需删减

## 二、定位与边界

### 2.1 目标用户

**OpenNexum 专为个人开发者设计**，不是通用 multi-agent 平台。

适用场景：
- 个人独立工作，无需多人协作模式
- 项目代码量 ≤ 20 万行，单次开发周期 ≤ 1 个月
- 使用 OpenClaw 作为主控层，调度 Claude Code / Codex / Google CLI 等 CLI agent
- 本机单机运行（Mac Mini），使用 tmux 管理 agent 进程

### 2.2 任务类型

OpenNexum 支持三类工作，共享同一套编排机制：

| 类型 | 示例 |
|------|------|
| **开发类** | 功能开发、Bug 修复、重构、测试、部署 |
| **日常任务类** | 数据整理、研究报告、信息汇总、文档翻译 |
| **创作类** | 写作、内容生成、脚本创作、设计描述 |

### 2.3 非目标

- 不支持多人团队协作
- 不支持分布式多机部署
- 不追求"通用万能"，有明确的用户边界

## 三、核心设计

### 3.1 Contract-Driven 执行

所有工作在启动前必须建立 **Task Contract**：在任何内容被生成之前，先对齐"什么叫完成"。

Contract 定义：
- **Scope**：要做什么、改哪些文件
- **Deliverables**：具体交付物清单
- **Eval Strategy**：用什么方式验证
- **Criteria**：逐条可验证的通过标准

Contract 是 generator 和 evaluator 共同可见的完成定义，不是 orchestrator 单方面写的 prompt。

### 3.2 Generator ≠ Evaluator（GAN 原则）

执行 agent（generator）和评估 agent（evaluator）严格分离：

- generator 专注完成任务
- evaluator 独立验证，不参与执行
- evaluator 被显式调校为"怀疑主义"：任务是找问题，而非确认没问题
- fail 时必须给出具体文件路径、函数名、期望行为 vs 实际行为，不允许模糊结论

### 3.3 知识持续沉淀

**对 agent 来说，context 里看不到的信息等于不存在。**

OpenNexum 通过两层结构确保知识可见、可演化：

**项目层**：每次任务完成后，agent 自动将遇到的问题和发现写入 `docs/lessons/`；harvest 流水线定期将 lessons 整理回 `AGENTS.md`，让下一个任务的 agent 能看到。

**Skill 层（OpenClaw）**：跨项目的通用规范和反模式库存储在 skill 内部；项目初始化时自动注入到项目结构中；gardener agent 定期将多个项目的经验蒸馏回 skill 层。

### 3.4 渐进式披露（Progressive Disclosure）

AGENTS.md 是目录，不是说明书，保持约 100 行。

> "给 agent 一张地图，而不是一本 1000 页的手册"

详细内容通过 AGENTS.md 的索引按需加载。过多的指导在上下文中会失去效果，甚至造成干扰。

### 3.5 机械约束优于 Prompt 约束

对于不变的规则（提交格式、文件命名、架构层级），用 linter/CI 机械强制执行，error message 里直接写修复指令。规则一旦编码，立即对所有 agent 生效，无需反复在 prompt 中叮嘱。

## 四、系统架构

### 4.1 分层结构

```
┌───────────────────────────────────────────────────┐
│  人类（Telegram / 终端）                           │
│  提需求 · 审核 Contract · 处理升级问题             │
└──────────────────┬────────────────────────────────┘
                   │
┌──────────────────▼────────────────────────────────┐
│  OpenClaw — Orchestrator（编排层）                 │
│                                                    │
│  • 接收需求，制定 Task Contract                    │
│  • 决定 eval strategy，dispatch generator          │
│  • 处理 evaluator 结果（pass/fail 循环）           │
│  • 管理知识流：lesson 写入 → harvest → 同步        │
│  • Telegram 通知人类关键节点                        │
└──┬──────┬──────┬──────┬──────┬────────────────────┘
   │      │      │      │      │
┌──▼──┐┌──▼──┐┌──▼──┐┌──▼──┐┌──▼──────┐
│plan ││exec ││write││eval ││gardener │
│规划 ││执行 ││创作 ││评估 ││知识维护 │
└─────┘└─────┘└─────┘└─────┘└─────────┘
```

### 4.2 Agent 角色

| Agent | 模型 | 职责 |
|-------|------|------|
| `plan` | claude-opus | 规划、制定/审核 Contract、拆解任务 |
| `codex-1` | gpt-5.4 | 后端/算法/脚本/内部工具执行 |
| `cc-frontend` | claude-sonnet | 外部用户可见的前端 UI |
| `cc-writer` | claude-sonnet | 写作、创作、文档类任务 |
| `eval` | claude-sonnet | 独立评估，不参与执行 |
| `gardener` | claude-sonnet | 定期 harvest lessons，维护知识库 |

**模型路由原则：**
- 后端/算法/内部工具 → Codex（token 经济）
- 外部用户可见 UI → Claude（质量优先）
- 写作/创作 → Claude（语言能力）
- 规划/评估 → Claude（批判性思维）

### 4.3 执行容器

使用 **tmux** 作为 agent 进程容器：
- 轻量，单机无额外依赖
- 随时 attach 查看 agent 实时状态
- 支持多 session 并行
- 通过 dispatch 脚本统一管理生命周期

## 五、Task Contract

### 5.1 Contract 格式

```yaml
# docs/nexum/contracts/TASK-001.yaml
id: TASK-001
name: "实现用户登录页面"
type: coding          # coding | writing | analysis | creative
created_at: "2026-03-27T09:00:00Z"

scope:
  files:
    - src/pages/login.tsx
    - src/api/auth.ts
  boundaries:
    - "不修改 src/components/ui/ 下的基础组件"
    - "不修改数据库 schema"

deliverables:
  - "登录表单，支持邮箱 + 密码"
  - "调用 POST /api/auth/login"
  - "错误状态展示"
  - "登录成功跳转 /dashboard"

eval_strategy:
  type: e2e
  tools: [playwright]
  criteria:
    - id: C1
      desc: "表单提交后正确调用 /api/auth/login"
      method: "playwright: intercept network request"
      threshold: pass
    - id: C2
      desc: "错误密码时展示错误提示"
      method: "playwright: check error element visible"
      threshold: pass
    - id: C3
      desc: "登录成功后 URL 变为 /dashboard"
      method: "playwright: check URL after submit"
      threshold: pass

generator: codex-1
evaluator: eval
max_iterations: 3
depends_on: []
```

### 5.2 Eval Strategy 选择矩阵

| 任务特征 | 推荐策略 |
|----------|----------|
| 纯算法/逻辑，有明确输入输出 | `unit` |
| 涉及数据库/API 集成 | `integration` |
| 有前端 UI，用户可交互 | `e2e`（Playwright） |
| 架构/代码质量/安全性 | `review` |
| 多维度复杂功能 | `composite`（组合） |
| 写作/创作类 | `review`（标准化评分） |
| 日常任务/研究报告 | `review`（完成度核查） |

### 5.3 Contract 制定流程

```
orchestrator 起草 contract draft
    ↓
（如需代码探索）plan agent 审核并补充技术细节
    ↓
orchestrator 最终确认，写入 contracts/TASK-ID.yaml
    ↓
dispatch generator
    ↓
generator 完成后，evaluator 拿 contract 逐条验证
    ↓
全部 pass → mark done
有 fail → feedback 回 generator，进入下一轮
超过 max_iterations → escalate to human
```

### 5.4 Evaluator 调校要求

evaluator prompt 必须包含：
1. **怀疑主义指令**："你的工作是找问题，不是夸奖输出"
2. **逐条验证**：对每条 criterion 必须给出 pass/fail + 具体证据，不允许整体判断
3. **可操作的 fail 反馈**：包含文件路径、函数名、期望行为 vs 实际行为
4. **禁止模糊通过**："整体看起来没问题"不构成通过标准

## 六、知识管理

### 6.1 Lesson 格式

agent 在完成任务过程中，遇到非预见问题或值得记录的模式时，写入：

```markdown
<!-- docs/lessons/YYYY-MM-DD-TASK-ID.md -->
## Lesson: [简短标题]
- trigger: [什么情况触发了这个问题]
- wrong: [错误做法或错误假设]
- right: [正确做法]
- affected: [具体文件/函数/API]        ← 有此字段 = 技术层 lesson
- agent_note: [orchestrator 下次注意]  ← 有此字段 = 流程层 lesson
- tags: [pg, auth, typescript, ...]
```

`affected` 和 `agent_note` 互斥：
- `affected` 有代码路径 → **技术层**，最终进入 AGENTS.md
- `agent_note` 描述 orchestrator 行为 → **流程层**，最终进入 skill 内置知识

### 6.2 Harvest 流水线

```
task 完成
  → on-complete 触发 harvest
      ├── 读新 lesson 文件
      ├── 与现有内容去重（关键词 + 标签匹配）
      ├── 互斥检查（是否与已有规范冲突）
      └── 根据结果分流：
          ├── 纯新增 + 无冲突 → 自动合并 + Telegram 通知"已更新"
          └── 有冲突 or 有歧义 → 生成 diff + Telegram 通知，等人工确认
```

### 6.3 知识层级

| 层级 | 存储位置 | 内容 | 更新方式 |
|------|----------|------|----------|
| 技术层 | `project/docs/lessons/` → `AGENTS.md` | 代码规范、反模式、踩坑 | harvest 自动同步 |
| 流程层 | `skill/references/lessons/` | orchestrator 行为优化 | harvest 更新 skill |

### 6.4 Gardener Agent

定期（手动触发或按周 cron）运行：
- 扫描所有 lesson 文件，识别过时条目
- 合并重复/相似内容
- 更新 AGENTS.md 技术层规范
- 输出变更摘要，人工审核后提交

## 七、项目初始化

### 7.1 命令

```bash
nexum init --project <name> --tech "<stack>" --type <coding|writing|mixed>
```

### 7.2 执行内容

1. 创建标准目录结构
2. 从 skill 模板生成 `AGENTS.md`（≈100 行目录 + 通用工程规范）
3. 生成 `CLAUDE.md`（Claude Code 专属约定）
4. 从 skill 反模式库提取与 tech stack 匹配的 lessons，注入 AGENTS.md
5. 初始化 `docs/` 目录（design / lessons / nexum/contracts）
6. 初始化 `nexum/active-tasks.json`
7. 安装 git hooks

### 7.3 标准项目结构

```
project/
├── AGENTS.md              ← 目录，≈100 行，索引指向 docs/
├── CLAUDE.md              ← Claude Code 专属约定
├── ARCHITECTURE.md        ← 架构概览（首次由 plan agent 生成）
├── docs/
│   ├── design/            ← 技术设计文档
│   ├── lessons/           ← 踩坑记录（agent 自动写入）
│   └── nexum/
│       └── contracts/     ← Task Contract 存档
└── nexum/
    ├── active-tasks.json  ← 当前批次任务状态
    ├── agent-pool.json    ← agent 状态
    └── history/           ← 已归档批次
```

### 7.4 AGENTS.md 内置学习协议

生成的 AGENTS.md 必须包含：

```markdown
## 错误学习协议

遇到非预见错误或值得记录的模式时：

1. 修复或完成任务
2. 写入 docs/lessons/YYYY-MM-DD-<TASK-ID>.md（使用标准模板）
3. 不要直接修改 AGENTS.md — 由 gardener 统一维护

这是强制要求，不是可选项。
```

## 八、任务生命周期

### 8.1 状态机

```
pending ──→ running ──→ evaluating ──→ done
              │               │
              ↓               ↓
            failed ←── eval_fail（未超次数）
              │
              ↓（超过 max_iterations）
           escalated → 通知人类
```

blocked：depends_on 未满足时的等待态，自动转 pending。

### 8.2 完整执行循环

```
1. orchestrator 制定 Task Contract
2. 注册任务（active-tasks.json）
3. dispatch generator
4. generator 执行，git commit
5. post-commit hook → push + 写信号
6. on-complete：
   a. 更新任务状态
   b. 触发 harvest（lesson 检查）
   c. dispatch evaluator（根据 eval_strategy）
   d. 唤醒 orchestrator
7. orchestrator 处理 eval 结果：
   - pass → mark done → 解锁下游 → dispatch 下一个
   - fail → feedback 回 generator → 进入下一轮
   - 超过 max_iterations → escalate
8. 所有任务 done → 汇报 + 触发 gardener
```

### 8.3 并行安全规则

file scope 不重叠的任务可并行 dispatch。dispatch 前检查：
- 新任务 scope 与所有 running 任务 scope 无重叠 → 可并行
- 有重叠 → 等待，或通过 `depends_on` 串行化

## 九、通知与可见性

### 9.1 通知分级

| 级别 | 触发条件 | 方式 |
|------|----------|------|
| 🔴 立即 | 超过 max_iterations；需人工决策；涉及 secrets/env | Telegram 立即推送 |
| 🟡 批次 | 全部任务完成；lesson harvest 冲突待审核 | Telegram |
| 🟢 静默 | 常规重试；agent 内部问答；routine fix | 只记录日志 |

### 9.2 Dispatch 卡片

每次 dispatch 发送：
```
🚀 Dispatched [TASK-ID] → [SESSION]
┣ 📋 Contract: docs/nexum/contracts/TASK-ID.yaml
┣ ⏰ Started:  HH:MM:SS
┣ 🤖 Model:   [模型名]（[reasoning level]）
┣ 📝 Task:    [一行描述]
┗ 🧪 Eval:    [eval_strategy type]
```

## 十、开发路线图

### Phase 1 — 合约基础（MVP）
- Task Contract YAML 规范
- `nexum init` 命令
- Contract-based dispatch
- Evaluator agent（review 模式）
- Lesson 写入协议

### Phase 2 — 验证能力
- Playwright e2e eval
- Unit / integration test eval
- composite eval strategy

### Phase 3 — 知识系统
- Harvest 流水线（去重 + 互斥检查 + 自动合并）
- 技术层 vs 流程层自动分流
- Gardener agent

### Phase 4 — 任务类型扩展
- Writing / Creative Contract 规范
- 写作类评分标准（design quality / originality）
- Analysis 任务支持

### Phase 5 — 工程加固
- linter / CI 架构约束
- Harvest 语义去重（embedding-based）
- 可靠性与性能优化

## 十一、待确认的开放问题

1. **Contract 格式**：YAML 可读性好但有缩进陷阱，是否用 YAML？或 YAML 写、JSON 存？

2. **Playwright eval 运行环境**：app 需在本地运行。是否要求所有 e2e 任务必须提供 `local_url`？

3. **Gardener 触发机制**：初期手动触发，还是每批次完成后自动触发？

4. **ClawHub 发布名**：`open-nexum` 还是 `opennexum`？

5. **现有项目迁移**：建议新项目直接使用 OpenNexum，旧项目按需迁移，不强制。
