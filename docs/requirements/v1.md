## 一、背景与动机

### 1.1 从 ayao-workflow-agent 到 OpenNexum

OpenNexum 是 ayao-workflow-agent（以下称 swarm v1）的继承者与重构。swarm v1 构建了基本可用的多 agent 编排能力，但在实际使用中暴露出几个根本性问题：

**问题一：知识孤岛**  
Claude 犯过的错误不能自动传达给 Codex，Codex 的教训也无法同步到下一个项目。每个 agent 在自己的上下文气泡里工作，没有跨 agent、跨项目的知识共享机制。

**问题二：验收标准形同虚设**  
`review_level=full/scan/skip` 是静态手工标注，review 本质上是"AI 看了一遍代码"，没有合约、没有可验证的完成标准、没有真实的功能验证。结果是：代码看起来没问题，但实际跑起来有 bug。

**问题三：调度层过薄**  
orchestrator（AI）的决策逻辑完全在 session 的"脑子"里——换个 session 就失忆，遇到复杂依赖关系就容易出错。决策规则没有代码化、没有可审计的记录。

**问题四：工程基础脆弱**  
dispatch 链路由 shell 脚本驱动，JSON 操作靠 python3 inline，任何脚本崩溃都可能产生幽灵任务。系统可靠性依赖脚本质量，维护成本高。

### 1.2 参考来源

本设计受以下两篇文章启发：

- **OpenAI Engineering：在智能体优先的世界中利用 Codex**  
  核心洞见：repo 是唯一 source of truth；AGENTS.md 是目录而非百科全书；架构约束靠机械强制（linter/CI）而非 prompt；知识必须存在 repo 里对 agent 才可见。

- **Anthropic Engineering：Harness design for long-running application development**  
  核心洞见：Generator + Evaluator 分离（GAN 思想）；Sprint Contract 在编码前先对齐"什么叫完成"；evaluator 用 Playwright 真实跑应用而非看代码；harness 每个组件都是对模型能力不足的假设，随模型进化可删减。

## 二、定位与边界

### 2.1 目标用户

**OpenNexum 是为"单人指挥多 agent"场景设计的编排框架**，不是通用 multi-agent 平台。

典型用户画像：
- 个人开发者，独立工作，不需要多人协作模式
- 项目代码量 < 20 万行，开发周期 < 1 个月
- 使用 OpenClaw 作为主控层，调度 Claude Code / Codex / Google CLI 等 CLI-based agent
- 使用 tmux 作为 agent 容器（轻量、可随时介入、支持多 session 并行）

### 2.2 核心扩展：不只是写代码

swarm v1 专注于 coding 任务。OpenNexum 扩展到三类工作：

| 类型 | 示例 |
|------|------|
| **开发类** | 功能开发、Bug 修复、重构、部署 |
| **日常任务类** | 数据整理、文档翻译、研究报告、信息汇总 |
| **创作类** | 写作、设计稿文字描述、内容生成、脚本创作 |

三类任务共享同一套 contract-driven 编排机制，但 evaluator 策略不同（见第五章）。

### 2.3 非目标

- 不支持多人/团队协作模式
- 不支持分布式多机部署（单机 Mac Mini）
- 不追求"通用万能"，宁可有边界也要有深度

## 三、核心设计哲学

### 3.1 Contract-Driven 执行

**这是 OpenNexum 与 swarm v1 最根本的区别。**

swarm v1 的模式：
```
需求 → 写 prompt → 执行 → (maybe) review → done
```

OpenNexum 的模式：
```
需求 → 制定 Contract → 执行 → 验证 Contract → done
```

**Task Contract** 是系统的核心概念。在任何代码/内容被生成之前，必须先确定：
- 要做什么（scope + deliverables）
- 改哪些文件（file boundaries）
- 怎么验证（eval strategy）
- 通过标准是什么（具体可验证条件）

Contract 是 generator 和 evaluator 共同可见的"完成定义"，不是 orchestrator 单方面写的 prompt。

### 3.2 Generator ≠ Evaluator

来自 GAN 思想：做的人不评判自己的输出。

- 执行 agent（generator）专注完成任务
- 评估 agent（evaluator）独立验证，不参与执行
- evaluator 被显式调校为"怀疑主义"——倾向于找问题，不是找夸奖的理由

### 3.3 知识即制度记忆

**对 agent 来说，它在 context 里看不到的东西就不存在。**

OpenNexum 通过三层结构保证知识可见：

```
Skill 层（OpenClaw）
  ├── 工程规范模板        ← 跨项目通用
  ├── 反模式库            ← 历史踩坑，新项目自动注入
  └── 流程层 lessons      ← orchestrator 自身的行为优化

Project 层
  ├── AGENTS.md           ← 由 skill 生成 + 项目特定内容
  ├── CLAUDE.md           ← Claude Code 专属约定
  └── docs/
      ├── lessons/        ← 本项目踩的坑（agent 自动写入）
      ├── design/         ← 设计文档
      └── nexum/          ← task contracts、dispatch 记录
```

### 3.4 渐进式披露（Progressive Disclosure）

AGENTS.md ≈ 100 行，是目录，不是说明书。

> "给 agent 一张地图，而不是一本 1000 页的手册"

详细内容通过 AGENTS.md 的索引按需加载。过多的指导 = 无效指导。

### 3.5 机械约束优于 Prompt 约束

对于不变的规则（架构层级、文件命名、提交格式），用 linter/CI 机械强制，error message 里直接写修复指令，而不是靠 prompt 反复叮嘱。

## 四、系统架构

### 4.1 分层结构

```
┌─────────────────────────────────────────────────────┐
│  人类（Telegram / 终端）                             │
│  • 提需求，审核 contract，处理升级                   │
└────────────────────┬────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────┐
│  OpenClaw 编排层（Orchestrator）                     │
│                                                      │
│  职责：                                              │
│  • 接收需求，制定/协商 Task Contract                 │
│  • 根据任务性质决定 eval strategy                   │
│  • dispatch generator agent                         │
│  • 触发 evaluator，处理 pass/fail 循环              │
│  • 管理知识流：lessons 写入、harvest、同步           │
│  • Telegram 通知人类关键节点                         │
└──┬──────┬──────┬──────┬──────┬──────────────────────┘
   │      │      │      │      │
   ▼      ▼      ▼      ▼      ▼
┌──────┐┌──────┐┌──────┐┌──────┐┌──────────┐
│plan  ││codex ││cc-   ││eval  ││gardener  │
│agent ││/-cc  ││front ││agent ││agent     │
│规划  ││执行  ││前端  ││评估  ││知识维护  │
└──────┘└──────┘└──────┘└──────┘└──────────┘
```

### 4.2 Agent 角色

| Agent | 模型 | 职责 |
|-------|------|------|
| `plan` | claude-opus | 规划、制定 contract、拆解任务 |
| `codex-1` | gpt-5.4 | 后端/算法/脚本执行 |
| `cc-frontend` | claude-sonnet | 外部用户可见的前端 |
| `cc-writer` | claude-sonnet | 写作/创作/文档类任务 |
| `eval` | claude-sonnet | 独立评估，不参与执行 |
| `gardener` | claude-sonnet | 定期 harvest lessons，维护知识库 |

> 原则：
> - 后端/算法 → Codex（token 经济）
> - 内部 UI/admin → Codex（token 经济）  
> - 外部用户可见 UI → Claude（质量优先）
> - 创作/写作 → Claude（语言能力）
> - 所有 eval → Claude（批判性思维）

### 4.3 执行容器

继续使用 **tmux** 作为 agent 容器，理由：
- 轻量，单机无额外依赖
- 支持随时 attach 查看 agent 状态
- 已有成熟的 dispatch/on-complete 基础设施
- 后续可升级为更健壮的 process manager，但保持接口兼容

## 五、Task Contract 规范

### 5.1 Contract 结构

```yaml
# nexum/contracts/TASK-001.yaml
id: TASK-001
name: "实现用户登录页面"
type: coding          # coding | writing | analysis | creative
created_at: "2026-03-27T09:00:00Z"

scope:
  files:
    - src/pages/login.tsx
    - src/api/auth.ts
  boundaries:
    - "不修改 src/components/ui/ 下的基础组件"
    - "不修改数据库 schema"

deliverables:
  - "登录表单，支持邮箱+密码"
  - "API 调用 POST /api/auth/login"
  - "错误状态展示"
  - "登录成功跳转 /dashboard"

eval_strategy:
  type: e2e             # unit | integration | e2e | review | composite
  tools: [playwright]
  criteria:
    - id: C1
      desc: "表单提交后正确调用 /api/auth/login"
      method: "playwright: intercept network request"
      threshold: pass
    - id: C2
      desc: "错误密码时展示错误提示"
      method: "playwright: check error element visible"
      threshold: pass
    - id: C3
      desc: "登录成功后 URL 变为 /dashboard"
      method: "playwright: check URL after submit"
      threshold: pass

generator: codex-1
evaluator: eval
max_iterations: 3
depends_on: []
```

### 5.2 Eval Strategy 选择矩阵

| 任务特征 | 推荐策略 |
|----------|----------|
| 纯算法/逻辑，有明确输入输出 | `unit` |
| 涉及数据库/API 集成 | `integration` |
| 有前端 UI，用户可交互 | `e2e` (Playwright) |
| 架构/代码质量/安全性 | `review` |
| 复杂功能，多个维度 | `composite`（组合） |
| 写作/创作类 | `review`（评分标准化） |
| 日常任务/研究报告 | `review`（完成度核查） |

### 5.3 Contract 协商流程

```
orchestrator 起草 contract draft
    ↓
plan agent 审核（如需代码探索）
    ↓
orchestrator 最终确认 contract
    ↓
dispatch generator
    ↓
generator 完成后，evaluator 拿 contract 验证
    ↓
pass → mark done
fail → feedback 回 generator，进入下一轮（max_iterations 限制）
```

### 5.4 Evaluator 调校原则

evaluator prompt 必须包含：
1. **怀疑主义指令**："你的工作是找问题，不是夸奖输出"
2. **criteria 逐条验证**：不能整体判断，必须对每条 criterion 给出 pass/fail + 证据
3. **可操作的 fail 反馈**：fail 时要包含具体文件路径、函数名、期望行为 vs 实际行为
4. **不允许模糊通过**："整体看起来没问题"不是通过标准

## 六、知识管理系统

### 6.1 Lesson 写入协议

每个 agent 在完成任务时，如果遇到非预见的问题或发现值得记录的模式，必须写入：

```markdown
<!-- docs/lessons/YYYY-MM-DD-TASK-ID.md -->
## Lesson: [简短标题]
- trigger: [什么情况触发了这个问题]
- wrong: [错误做法或错误假设]
- right: [正确做法]
- affected: [具体文件/函数/API]        ← 有此字段 = 技术层 lesson
- agent_note: [orchestrator 下次注意]  ← 有此字段 = 流程层 lesson
- tags: [pg, auth, typescript, ...]
```

`affected` 和 `agent_note` 互斥：有代码路径 → 技术层；有 orchestrator 行为描述 → 流程层。

### 6.2 Harvest 流水线

```
task 完成
  → on-complete.sh 触发 harvest
      ├── 读新 lesson 文件
      ├── 与现有内容语义去重（关键词 + 标签匹配）
      ├── 互斥检查（新 lesson 与已有规范是否冲突）
      └── 生成 diff
          ├── 纯新增 + 无冲突 → 自动合并 + Telegram 通知"已更新"
          └── 有冲突 or 不确定 → Telegram 发 diff，等人工确认
```

### 6.3 两层存储

| 层级 | 存储位置 | 内容 | 更新方式 |
|------|----------|------|----------|
| 技术层 | `project/docs/lessons/` + `AGENTS.md` | 具体代码规范、反模式 | harvest 自动同步到 AGENTS.md |
| 流程层 | `skill/references/lessons/` | orchestrator 行为优化 | harvest 更新 skill 内置知识 |

### 6.4 Gardener Agent

每周（或手动触发）运行一次：
- 扫描所有 lessons，识别已经过时的条目
- 识别重复/相似条目，合并
- 更新 AGENTS.md 中的技术层规范
- 输出变更摘要，人工审核后提交

## 七、项目初始化

### 7.1 init 命令

```bash
nexum init --project <name> --tech "<stack>" --type <coding|writing|mixed>
```

执行内容：
1. 创建标准目录结构
2. 从 skill 模板生成 `AGENTS.md`（100 行，含通用工程规范 + 目录索引）
3. 生成 `CLAUDE.md`（Claude Code 专属约定）
4. 从 skill 反模式库提取与该 tech stack 相关的 lessons，注入 AGENTS.md
5. 创建 `docs/` 结构
6. 初始化 `nexum/` 目录（contracts、active-tasks.json）
7. 安装 git hooks

### 7.2 生成的项目结构

```
project/
├── AGENTS.md              ← 地图，≈100行，指向 docs/
├── CLAUDE.md              ← Claude Code 专属约定
├── ARCHITECTURE.md        ← 架构概览（首次由 plan agent 生成）
├── docs/
│   ├── design/            ← 技术设计文档
│   ├── lessons/           ← 踩坑记录（agent 自动写入）
│   └── nexum/             ← task contracts 存档
│       └── contracts/
└── nexum/
    ├── active-tasks.json  ← 当前批次任务状态
    ├── agent-pool.json    ← agent 状态
    └── history/           ← 已归档批次
```

### 7.3 AGENTS.md 内置的学习协议

生成的 AGENTS.md 必须包含此段：

```markdown
## 错误学习协议

遇到任何未预见的错误、发现值得记录的模式时：

1. 修复或完成任务
2. 在 docs/lessons/YYYY-MM-DD-<TASK-ID>.md 追加一条记录
3. 使用标准格式（见 docs/lessons/TEMPLATE.md）
4. 不要修改 AGENTS.md 本身 — 由 gardener 统一维护

这是强制要求，不是可选项。
```

## 八、任务生命周期

### 8.1 状态机

```
pending → running → done
              ↓         ↑
           failed → retrying (max 3次)
              ↓
           escalated（通知人类）
```

blocked 状态：depends_on 未完成时的等待态，dependencies 满足后自动转 pending。

### 8.2 完整执行循环

```
1. orchestrator 制定 contract
2. 注册任务到 active-tasks.json
3. dispatch generator（dispatch.sh）
4. generator 执行，git commit
5. post-commit hook → push + 写信号
6. on-complete.sh：
   a. 更新任务状态
   b. 触发 harvest（lessons 检查）
   c. 触发 evaluator（根据 eval_strategy）
   d. openclaw system event → 唤醒 orchestrator
7. orchestrator 处理 eval 结果：
   - pass → mark done → 解锁下游 → dispatch 下一个
   - fail → feedback 回 generator → 进入下一轮
   - 超过 max_iterations → escalate to human
8. 所有任务 done → 汇报 + 触发 gardener
```

### 8.3 并行安全

file scope 不重叠的任务可以并行 dispatch。orchestrator 在 dispatch 前检查：
- 读取所有 running 任务的 file scope
- 新任务 scope 与任何 running 任务不重叠 → 可并行
- 有重叠 → 等待，或调整 depends_on

## 九、通知与可见性

### 9.1 通知分级

| 级别 | 触发条件 | 通知方式 |
|------|----------|----------|
| 🔴 立即 | 任务超过 max_iterations；需要人工决策；secrets/env 相关 | Telegram 立即 |
| 🟡 批次 | 所有任务完成；里程碑达成；lesson harvest 冲突待审核 | Telegram |
| 🟢 静默 | 常规重试；agent 问答；routine fixes | 只记录，不通知 |

### 9.2 Dispatch 卡片格式

每次 dispatch 必须发送（verbose 模式）：
```
🚀 Dispatched [TASK-ID] → [SESSION]
┣ 📋 Contract: [contract 文件路径]
┣ ⏰ Started: [HH:MM:SS]
┣ 🤖 Model: [模型名] ([reasoning level])
┣ 📝 Task: [一行描述]
┗ 🧪 Eval: [eval_strategy type]
```

## 十、从 swarm v1 的继承与差异

### 10.1 保留

- tmux 作为 agent 容器
- dispatch.sh / on-complete.sh 机制（升级，不推翻）
- active-tasks.json 任务注册
- git hook 事件驱动
- 编排层与执行层分离原则
- 主干开发（main branch only）
- 原子 commit

### 10.2 升级

| 维度 | swarm v1 | OpenNexum |
|------|----------|-----------|
| 任务完成标准 | prompt 里描述 | 结构化 Contract YAML |
| Eval | review_level 静态分级 | 动态 eval_strategy + Contract 验证 |
| 知识共享 | 无 | lessons 系统 + harvest 流水线 |
| 项目启动 | 手动 | `nexum init` 自动生成规范 |
| 任务类型 | 仅 coding | coding + writing + creative + analysis |
| Evaluator | 同 generator，review 代码 | 独立 agent，执行 Contract criteria |

### 10.3 废弃

- `review_level: full/scan/skip` → 替换为 `eval_strategy`
- 手动写 active-tasks.json → `nexum register` 命令
- SKILL.md 里的大段 inline 说明 → 移入结构化文档，SKILL.md 保持简洁

## 十一、开发路线图

### Phase 1：基础合约系统（MVP）
- Contract YAML 规范定义
- `nexum init` 命令（生成项目结构 + AGENTS.md）
- Contract-based dispatch（dispatch.sh 升级）
- 基础 evaluator agent（review 模式，先不做 e2e）
- Lesson 写入协议（模板 + AGENTS.md 内置指令）

### Phase 2：验证能力
- Playwright eval 能力（e2e 模式）
- Unit test eval（运行 test suite，解析结果）
- Integration test eval
- composite eval strategy

### Phase 3：知识系统
- Harvest 流水线（去重、互斥检查、自动合并）
- 技术层 vs 流程层 lesson 自动分流
- Gardener agent（周期性知识维护）

### Phase 4：任务类型扩展
- Writing/Creative 任务的 Contract 规范
- eval 标准化评分（design quality / originality 等）
- Analysis 任务支持

### Phase 5：工程加固
- linter/CI 架构约束
- harvest 语义去重升级（embedding-based）
- 性能与可靠性优化

## 十二、待讨论的开放问题

1. **Contract YAML vs JSON**：YAML 可读性好但有缩进陷阱，JSON 更机械安全。初期用 YAML，工具链用 js-yaml 解析？

2. **Playwright eval 的运行环境**：本机直接跑（依赖 app 在本地运行）还是需要 CI 环境？PolyGo 这类项目有 localhost 访问，应该可行。

3. **Gardener 触发机制**：每周定时（cron）还是每批次完成后触发？前期建议手动触发，避免意外。

4. **skill 名称**：ClawHub 上发布名是 `open-nexum` 还是 `opennexum`？

5. **向后兼容**：ayao-workflow-agent 里的现有项目（PolyGo）如何迁移到 OpenNexum？建议新项目直接用 OpenNexum，旧项目按需迁移。
