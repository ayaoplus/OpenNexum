## 一、概述

OpenNexum 是一个运行在 OpenClaw 上的多 agent 编排框架，专为**单人指挥、多 agent 并行执行**的工作模式设计。

它的核心理念是：**合约驱动执行，知识持续沉淀**。每项工作在启动前先确立可验证的完成标准（Contract），完成后自动将经验写回系统供后续复用。随着时间推移，系统变得越来越了解你的项目和工作习惯。

### 设计背景

OpenNexum 在早期 agent 编排实践的基础上，吸收了 OpenAI 和 Anthropic 在智能体工程上的核心经验：

- **Repo 是唯一的 source of truth**：agent 看不到的信息等于不存在，所有知识必须结构化存储在项目里
- **Generator 与 Evaluator 分离**：做的人不应该评判自己的输出，独立 evaluator 是质量的真正保障
- **harness 组件应当可裁剪**：每个组件都是对模型能力不足的补偿假设，随模型进化可按需删减

## 二、定位与边界

### 2.1 目标用户

**OpenNexum 专为个人开发者设计**，不是通用 multi-agent 平台。

适用场景：
- 个人独立工作，无需多人协作模式
- 项目代码量 ≤ 20 万行，单次开发周期 ≤ 1 个月
- 使用 OpenClaw 作为主控层，调度 Claude Code / Codex / Google CLI 等 CLI agent
- 本机单机运行（Mac Mini），使用 tmux 管理 agent 进程

### 2.2 任务类型

OpenNexum 支持三类工作，共享同一套编排机制：

| 类型 | type 枚举值 | 示例 | 默认 Agent |
|------|------------|------|-----------|
| **开发类** | `coding` | 功能开发、Bug 修复、重构、测试、部署 | `codex-1` / `cc-frontend` |
| **日常任务类** | `task` | 数据整理、研究报告、信息汇总、文档翻译 | `cc-writer` |
| **创作类** | `creative` | 写作、内容生成、脚本创作、设计描述 | `cc-writer` |

> 三种类型均已实现。`dispatch.sh` 根据 `contract.type` 自动路由 generator prompt 模板：`coding` 使用 `prompt-generator-coding.md`，`creative`/`task` 使用 `prompt-generator-writing.md`。

### 2.3 非目标

- 不支持多人团队协作
- 不支持分布式多机部署
- 不追求"通用万能"，有明确的用户边界

## 三、核心设计

### 3.1 Contract-Driven 执行

所有工作在启动前必须建立 **Task Contract**：在任何内容被生成之前，先对齐"什么叫完成"。

Contract 定义：
- **Scope**：要做什么、改哪些文件、明确禁止修改的范围
- **Deliverables**：具体交付物清单
- **Eval Strategy**：用什么方式验证
- **Criteria**：逐条可验证的通过标准

Contract 是 generator 和 evaluator 共同可见的完成定义，不是 orchestrator 单方面写的 prompt。**Contract 一旦创建即为静态，不在执行过程中修改**（运行态信息存储在 active-tasks.json）。

### 3.2 Generator ≠ Evaluator（GAN 原则）

执行 agent（generator）和评估 agent（evaluator）严格分离：

- generator 专注完成任务
- evaluator 独立验证，不参与执行
- evaluator 被显式调校为"怀疑主义"：任务是找问题，而非确认没问题
- fail 时必须给出具体文件路径、函数名、期望行为 vs 实际行为，不允许模糊结论

### 3.3 知识持续沉淀

**对 agent 来说，context 里看不到的信息等于不存在。**

OpenNexum 通过两层结构确保知识可见、可演化：

**项目层**：每次任务完成后，agent 自动将遇到的问题和发现写入 `docs/lessons/`；harvest 流水线（`scripts/harvest.sh`）将 lessons 整理回 `AGENTS.md` 的 `nexum:lessons` 区间，让下一个任务的 agent 能看到。

**Skill 层（OpenClaw）**：跨项目的通用规范和反模式库存储在 skill 内部；项目初始化时自动注入到项目结构中；gardener agent（`scripts/dispatch-gardener.sh`）定期将多个项目的经验蒸馏回 skill 层（`references/lessons/`）。

### 3.4 渐进式披露（Progressive Disclosure）

AGENTS.md 是目录，不是说明书，保持约 100 行。

> "给 agent 一张地图，而不是一本 1000 页的手册"

详细内容通过 AGENTS.md 的索引按需加载。Gardener agent 的核心职责之一是维护 AGENTS.md 在 ~100 行以内——低频使用的条目会被降级到 `docs/lessons/` 详细文档，AGENTS.md 只保留索引。

### 3.5 机械约束优于 Prompt 约束

对于不变的规则（提交格式、文件命名、架构层级），用 linter/CI 机械强制执行，error message 里直接写修复指令。规则一旦编码，立即对所有 agent 生效，无需反复在 prompt 中叮嘱。

`nexum-lint.sh` 是这一原则的核心实现：Contract YAML 通过 pre-commit hook 自动校验，不合规的 Contract 无法提交。

## 四、系统架构

### 4.1 分层结构

```
┌───────────────────────────────────────────────────────┐
│  人类（Telegram / 终端）                               │
│  提需求 · 审核 Contract · 处理升级问题                 │
└────────────────────┬──────────────────────────────────┘
                     │
┌────────────────────▼──────────────────────────────────┐
│  OpenClaw — Orchestrator（编排层）                     │
│                                                        │
│  • 接收需求，制定 Task Contract                        │
│  • 决定 eval strategy，dispatch generator              │
│  • 处理 evaluator 结果（pass/fail 循环）               │
│  • 管理知识流：lesson 写入 → harvest → 同步            │
│  • Telegram 通知人类关键节点                            │
└──┬──────┬──────┬──────┬──────┬──────┬──────┬──────────┘
   │      │      │      │      │      │      │
┌──▼──┐┌──▼──┐┌──▼───┐┌─▼──┐┌──▼──┐┌──▼───┐┌─▼──────┐
│plan ││cod  ││cod   ││cc- ││cc-  ││eval  ││gardener│
│     ││ex-1 ││ex-   ││fr- ││wri- ││      ││        │
│规划 ││后端 ││front ││ont ││ter  ││评估  ││知识维护│
│     ││     ││内部  ││外部││写作 ││      ││        │
└─────┘└─────┘└──────┘└────┘└─────┘└──────┘└────────┘
```

### 4.2 Agent 角色

| Agent | 默认模型 | 职责 |
|-------|---------|------|
| `plan` | claude-opus | 规划、制定/审核 Contract、拆解任务 |
| `codex-1` | gpt-5.4 | 后端、算法、脚本、部署 |
| `codex-frontend` | gpt-5.4 | 内部工具、admin 类前端（对样式无高要求） |
| `cc-frontend` | claude-sonnet | 外部用户可见的前端 UI（质量优先） |
| `cc-writer` | claude-sonnet | 写作、创作、文档类任务 |
| `eval` | gpt-5.4 | 独立评估，不参与执行；主观/创作任务可配置为 claude-sonnet |
| `gardener` | claude-sonnet | 定期 harvest lessons，维护知识库 |

**模型路由默认建议（可通过配置覆盖）：**

| 任务类型 | 默认 Agent | 理由 |
|----------|-----------|------|
| 后端 / 算法 / 脚本 | `codex-1` | token 经济 |
| 内部 admin / 后台 UI | `codex-frontend` | 样式要求低，节省 token |
| 外部用户可见 UI | `cc-frontend` | 质量优先 |
| 写作 / 创作 | `cc-writer` | 语言能力 |
| eval（客观验证） | `eval`（gpt-5.4） | 速度 + 经济 |
| eval（主观/创作） | `eval`（claude-sonnet） | 批判性思维，通过 `evaluator_model` 配置 |
| 规划 / Contract 制定 | `plan`（claude-opus） | 全局理解能力 |

> ⚠️ 以上为**建议默认值**。所有 agent 的模型和 CLI 工具均可在项目配置中独立覆盖，系统不强制绑定任何模型。

Skill 级默认值存储在 `skill/references/defaults.json`，项目级配置（`nexum/config.json`）覆盖其中任意字段。可通过 `nexum config --show` 查看当前生效的完整合并配置。

### 4.3 执行容器

使用 **tmux** 作为 agent 进程容器：
- 轻量，单机无额外依赖
- 随时 attach 查看 agent 实时状态
- 支持多 session 并行
- 通过 dispatch 脚本统一管理生命周期
- `nexum-init.sh` 自动创建标准 session：`nexum-plan`、`nexum-codex-1`、`nexum-codex-frontend`、`nexum-cc-frontend`、`nexum-eval`

### 4.4 AGENTS.md 与 CLAUDE.md 同步

项目根目录同时维护两个文件以兼容不同 CLI：

- `AGENTS.md`：Codex CLI 自动加载
- `CLAUDE.md`：Claude Code 自动加载

**`AGENTS.md` 是唯一 source of truth**，`CLAUDE.md` 由 dispatch.sh 在 force-commit 阶段显式同步（`cp AGENTS.md CLAUDE.md`）。git hook 中**不做任何 commit 操作**，避免递归触发。

> 规则：永远不要直接编辑 `CLAUDE.md`。

## 五、Task Contract

### 5.1 Coding 类 Contract 格式

```yaml
# docs/nexum/contracts/TASK-001.yaml
id: TASK-001                    # 格式：[A-Z]+-[0-9]+
name: "实现用户登录页面"
type: coding
created_at: "2026-03-27T09:00:00Z"

scope:
  files:
    - src/pages/login.tsx
    - src/api/auth.ts
  boundaries:
    - "不修改 src/components/ui/ 下的基础组件"
    - "不修改数据库 schema"
  conflicts_with: []            # 与哪些 task id 存在语义冲突，需串行

deliverables:
  - "登录表单，支持邮箱 + 密码"
  - "调用 POST /api/auth/login"
  - "错误状态展示"
  - "登录成功跳转 /dashboard"

eval_strategy:
  type: e2e
  tools: [playwright]
  local_url: "http://localhost:3000"   # e2e 必填
  criteria:
    - id: C1
      desc: "表单提交后正确调用 /api/auth/login"
      method: "playwright: intercept network request"
      threshold: pass
    - id: C2
      desc: "错误密码时展示错误提示"
      method: "playwright: check error element visible"
      threshold: pass
    - id: C3
      desc: "登录成功后 URL 变为 /dashboard"
      method: "playwright: check URL after submit"
      threshold: pass

generator: codex-frontend
evaluator: eval
max_iterations: 3
depends_on: []
```

### 5.2 Creative/Task 类 Contract 格式

```yaml
# docs/nexum/contracts/WRITE-001.yaml
id: WRITE-001
name: "撰写 PolyGo 产品介绍文档"
type: creative
created_at: "2026-03-27T09:00:00Z"

scope:
  files:
    - docs/product-intro.md
  boundaries:
    - "不修改其他文档"
  conflicts_with: []

deliverables:
  - "产品背景和核心功能介绍（500-800字）"
  - "面向技术用户，清晰描述系统架构"
  - "包含使用场景举例"

eval_strategy:
  type: review
  evaluator_model: claude-sonnet   # 创作类覆盖为 claude-sonnet
  criteria:
    - id: C1
      desc: "内容完整性：所有 deliverables 均已覆盖"
      method: "review: checklist"
      threshold: pass
    - id: C2
      desc: "表达清晰度"
      method: "review: score"
      threshold: score
      min_score: 7
    - id: C3
      desc: "原创性：无明显 AI 模板化痕迹"
      method: "review: score"
      threshold: score
      min_score: 6

generator: cc-writer
evaluator: eval
max_iterations: 3
depends_on: []
```

**评分制说明**：当 `threshold: score` 时，evaluator 对该 criterion 给出 1-10 分。`min_score` 指定通过门槛，未指定时默认为 6。

### 5.3 Eval Strategy 选择矩阵

| 任务特征 | 推荐策略 | type 值 | 执行方式 | 默认 Evaluator 模型 |
|----------|----------|---------|----------|-------------------|
| 纯算法/逻辑，有明确输入输出 | unit | `unit` | 运行项目 test suite，解析结果 | gpt-5.4 |
| 涉及数据库/API 集成 | integration | `integration` | 启动服务，调用 API，验证响应（`shell:`/`review:` method 前缀） | gpt-5.4 |
| 有前端 UI，用户可交互 | e2e | `e2e` | Playwright 驱动浏览器执行 criteria（需 `local_url`） | gpt-5.4 |
| 架构/代码质量/安全性 | review | `review` | evaluator 读代码，逐条 criteria 给 pass/fail | gpt-5.4 |
| 多维度复杂功能 | composite | `composite` | 多个 `sub_strategies` 顺序执行，合并 verdict；全部 pass 才 pass | gpt-5.4 |
| 写作/创作类 | review（评分制） | `review` | evaluator 读内容，criteria 用 `threshold: score` + `min_score` | claude-sonnet |
| 日常任务/研究报告 | review（完成度核查） | `review` | evaluator 核对 deliverables 是否齐全 | gpt-5.4 |

**e2e 评估**要求 Contract 中指定 `eval_strategy.local_url`（如 `http://localhost:3000`），或在项目配置中设置 `eval.default_local_url`。`dispatch-evaluator.sh` 使用专用 prompt 模板 `prompt-evaluator-e2e.md`。

**integration 评估**使用 `prompt-evaluator-integration.md` 模板，支持 `shell:` 和 `review:` 两种 method 前缀。

**composite 评估**要求 `eval_strategy.sub_strategies` 为非空数组，每个子策略独立定义 type 和 criteria。不允许嵌套 composite。使用 `prompt-evaluator-composite.md` 模板。

### 5.4 并行安全规则

**file scope 是并行的必要条件，但不是充分条件。**

允许并行的条件（同时满足）：
1. `scope.files` 集合与所有 running 任务无交集
2. `conflicts_with` 中未列出任何 running task id
3. 无语义依赖（不修改被其他 running 任务依赖的接口/类型）

当并行条件不满足时，通过 `depends_on` 串行化（而非运行时检测）。**Evaluator 是并行冲突的最终兜底**——eval 阶段会发现 scope 冲突导致的实际问题。

### 5.5 Contract 制定流程

```
orchestrator 起草 contract draft（含 conflicts_with 声明）
    ↓
（如需代码探索）plan agent 审核并补充技术细节
    ↓
orchestrator 最终确认，写入 contracts/TASK-ID.yaml
    ↓ ← pre-commit hook 执行 nexum-lint.sh 校验 Contract YAML
dispatch generator（同时记录 base_commit）
    ↓
generator 完成后，evaluator 拿 contract 逐条验证
    ↓
全部 pass → mark done
有 fail → feedback 回 generator，进入下一轮
超过 max_iterations → escalate to human
```

### 5.6 Evaluator 调校要求

evaluator prompt 必须包含：
1. **怀疑主义指令**："你的工作是找问题，不是夸奖输出"
2. **逐条验证**：对每条 criterion 必须给出 pass/fail + 具体证据，不允许整体判断
3. **可操作的 fail 反馈**：包含文件路径、函数名、期望行为 vs 实际行为
4. **禁止模糊通过**："整体看起来没问题"不构成通过标准

### 5.7 Contract 校验（nexum-lint）

`scripts/nexum-lint.sh` 提供 Contract 和项目结构的静态校验。

**Contract 检查项**（per YAML file）：

| # | 检查代码 | 说明 |
|---|---------|------|
| 1 | `CONTRACT_ID_INVALID` | id 格式必须匹配 `[A-Z]+-[0-9]+` |
| 2 | `CONTRACT_TYPE_INVALID` | type 必须为 `coding`/`task`/`creative` |
| 3 | `NAME_MISSING` | name 必须为非空字符串 |
| 4 | `CREATED_AT_MISSING` | created_at 必须为 ISO 8601 UTC 时间戳 |
| 5 | `MAX_ITERATIONS_INVALID` | max_iterations 必须为正整数 |
| 6 | `DEPENDS_ON_MISSING` | depends_on 必须为数组 |
| 7 | `SCOPE_FILES_EMPTY` | scope.files 至少包含一个文件 |
| 8 | `SCOPE_BOUNDARIES_MISSING` | scope.boundaries 必须为数组 |
| 9 | `SCOPE_CONFLICTS_MISSING` | scope.conflicts_with 必须为数组 |
| 10 | `DELIVERABLES_EMPTY` | deliverables 至少包含一个可验证交付物 |
| 11 | `EVAL_TYPE_INVALID` | eval_strategy.type 必须为有效枚举值 |
| 12 | `E2E_MISSING_LOCAL_URL` | e2e 类型必须指定 local_url |
| 13 | `CRITERIA_EMPTY` / `CRITERION_INCOMPLETE` | criteria 非空 + 每条必须有 id/desc/method/threshold |
| 14 | `COMPOSITE_NO_SUBS` | composite 类型必须指定 sub_strategies |
| 15 | `GENERATOR_MISSING` | generator 必须指定 agent |
| 16 | `EVALUATOR_MISSING` | evaluator 必须指定 agent |
| 17 | `TYPE_AGENT_MISMATCH` | coding 不应使用 cc-writer；creative/task 应使用 cc-writer |

**项目结构检查项**（5 项，级别为 warning）：

| # | 检查代码 | 说明 |
|---|---------|------|
| 1 | `AGENTS_MD_MISSING` | AGENTS.md 是否存在 |
| 2 | `AGENTS_MD_NO_MARKER` | AGENTS.md 是否包含 nexum:lessons 区间标记 |
| 3 | `ACTIVE_TASKS_MISSING` | nexum/active-tasks.json 是否存在 |
| 4 | `HOOKS_NOT_INSTALLED` | post-commit hook 是否已安装且可执行 |
| 5 | `EVAL_DIR_MISSING` | nexum/runtime/eval/ 目录是否存在 |

**使用方式**：
```bash
# 校验所有 Contract + 项目结构（默认模式）
nexum-lint.sh

# 校验单个 Contract
nexum-lint.sh --contract docs/nexum/contracts/TASK-001.yaml

# 仅校验项目结构
nexum-lint.sh --project .

# 不显示修复建议
nexum-lint.sh --no-fix-hints
```

**退出码**：0 = 无 error；1 = 有 error。warning 不影响退出码。修复建议默认显示，`--no-fix-hints` 可抑制。

## 六、知识管理

### 6.1 Lesson 格式

agent 在完成任务过程中，遇到非预见问题或值得记录的模式时，写入：

```markdown
<!-- docs/lessons/YYYY-MM-DD-TASK-ID.md -->
## Lesson: [简短标题]
- trigger: [什么情况触发了这个问题]
- wrong: [错误做法或错误假设]
- right: [正确做法]
- affected: [具体文件/函数/API]        ← 有此字段 = 技术层 lesson
- agent_note: [orchestrator 下次注意]  ← 有此字段 = 流程层 lesson
- tags: [pg, auth, typescript, ...]
```

`affected` 和 `agent_note` 互斥：
- `affected` 有代码路径 → **技术层**，最终进入 AGENTS.md
- `agent_note` 描述 orchestrator 行为 → **流程层**，最终进入 skill 内置知识（`references/lessons/`）
- 两字段同时存在或同时缺失 → 标记为 `invalid`，harvest 时报告

### 6.2 Harvest 流水线

`scripts/harvest.sh` 实现完整的 lesson 收割流水线：

```
触发时机：批次全部任务 done 且 harvest.auto_trigger: true 时，由 on-complete.sh 自动触发
    ├── 读取 docs/lessons/ 下所有 .md 文件（排除 TEMPLATE.md）
    ├── 解析 markdown bullet 字段为结构化数据
    ├── 按 mtime 对比 .harvest-state.json 中记录的 last_harvest_at，识别新增文件
    ├── 按字段分流：
    │   ├── affected 有值 → 技术层
    │   ├── agent_note 有值 → 流程层
    │   └── 两者都有或都无 → invalid，跳过并报告
    ├── 技术层处理：
    │   ├── 构建 TF-IDF 语料库（基于 right + trigger 字段的词条）
    │   ├── 去重：同一 affected path + cosine similarity > 0.65 → DUPLICATE
    │   ├── 矛盾检测：同一 trigger + similarity ≥ 0.55 + 否定词方向相反 → CONTRADICTION
    │   └── 无冲突的 lesson 写入 AGENTS.md nexum:lessons 区间
    ├── 流程层处理：
    │   └── 新增 lesson 镜像写入 skill/references/lessons/（按项目+路径命名）
    └── 根据结果分流：
        ├── 纯新增 + 无冲突 → 自动更新 AGENTS.md + Telegram 通知
        └── 有冲突 → Telegram 通知人工审核（含冲突详情和 diff）
```

**TF-IDF 去重实现**：纯 Python 标准库实现（math + re），不依赖 pip 包。计算方式为：

1. 对语料库所有 lesson 的 `right` + `trigger` 文本分词
2. 构建 IDF 向量：`log((N+1)/(df+1)) + 1.0`
3. 每条 lesson 计算 TF-IDF 向量
4. 两两计算 cosine similarity

**矛盾检测**：在词汇重叠度 ≥ 0.25 的前提下，如果一方包含否定词（not、never、avoid、don't 等）而另一方不包含，则认为存在矛盾。

**AGENTS.md 更新规则**：
- 只修改 `<!-- nexum:lessons:start -->` 和 `<!-- nexum:lessons:end -->` 之间的内容
- 更新前通过 `git blame` 检测区间是否有人工修改；如有，停止自动合并，通知人工处理
- 更新后自动 `git commit`（`--no-verify` 避免 hook 递归）

**状态持久化**：`nexum/.harvest-state.json` 记录 `last_batch_id`、`last_harvest_at`、`processed_files` 列表。

### 6.3 知识层级

| 层级 | 存储位置 | 内容 | 更新方式 |
|------|----------|------|----------|
| 技术层 | `project/docs/lessons/` → `AGENTS.md` nexum:lessons 区间 | 代码规范、反模式、踩坑 | harvest.sh 自动触发 |
| 流程层 | `skill/references/lessons/` | orchestrator 行为优化 | harvest.sh 镜像写入 |

### 6.4 Gardener Agent

**自动触发**（批次完成后）或手动触发（`scripts/dispatch-gardener.sh`）：

- Prompt 模板：`references/prompt-gardener.md`
- 扫描所有 lesson 文件，识别过时/失效条目（referenced 文件已不存在）
- 合并重复/相似内容
- 维护 AGENTS.md 总行数 ~100 行（低频条目降级）
- 输出变更摘要，人工审核后提交

## 七、项目初始化

### 7.1 命令

```bash
nexum-init.sh --project <name> [--tech "<stack>"] [--type <coding|mixed>]
```

`--type` 是项目维度的大类（`coding` = 纯开发项目；`mixed` = 开发 + 写作/分析混合），与 Contract 里的 task `type` 枚举值是不同维度的分类。

### 7.2 自动初始化

`dispatch.sh` 和 `dispatch-evaluator.sh` 在检测到 `nexum/active-tasks.json` 不存在时，会自动调用 `nexum-init.sh`：

```bash
# 自动推断项目名 = 目录名，类型 = coding
NEXUM_PROJECT_DIR="$PROJECT_DIR" nexum-init.sh --project "$(basename $PROJECT_DIR)" --type coding
```

无需人工手动执行初始化。已初始化的项目不会重复执行（`nexum-init.sh` 幂等：检测到 `active-tasks.json` 存在时静默退出）。

### 7.3 执行内容

1. 创建标准目录结构
2. 从 skill 模板（`references/agents-md-template.md`）渲染 `AGENTS.md`（≈100 行目录 + 通用工程规范）
3. 自动生成 `CLAUDE.md`（复制 AGENTS.md 内容，后续由 dispatch.sh 维护同步）
4. 复制 lesson 写入模板（`references/lesson-template.md` → `docs/lessons/TEMPLATE.md`）
5. 从 skill 级默认值（`references/defaults.json`）生成 `nexum/config.json`
6. 初始化 `nexum/active-tasks.json`（空批次，含 batch_id、project、timestamps）
7. 写入初始事件到 `nexum/events.jsonl`（`project_initialized` 事件）
8. 配置 `.gitignore`（添加 `nexum/runtime/screenshots/`）
9. 如无 `.git` 则 `git init`
10. 安装 git hooks（post-commit + pre-commit）
11. 创建 tmux sessions：`nexum-plan`、`nexum-codex-1`、`nexum-codex-frontend`、`nexum-cc-frontend`、`nexum-eval`

### 7.4 标准项目结构


## 八、任务生命周期

### 8.1 状态机

```
                    ┌─────────────────────────┐
                    ▼                         │（depends_on 重新 blocked）
blocked ──→ pending ──→ running ──→ evaluating ──→ done
                           │             │
                           ▼             ▼
                         failed ←── eval_fail（未超次数）
                           │
                           ▼（超过 max_iterations）
                        escalated ──→ 通知人类，等待决策
                           │
                           ▼（人类处理后）
                          done / cancelled

cancelled：任何状态均可转入（人类主动中止）
  → 对应 tmux session 被终止
  → 下游 blocked 任务转为 escalated（等人类决定是否继续）
```

`blocked`：depends_on 中存在未 done 的 task 时的等待态。所有依赖 done → 自动转 pending。若依赖 task 最终为 `escalated` / `cancelled` → 当前 blocked task 同样转为 `escalated`，通知人类决策。

### 8.2 完整执行循环

```
1. orchestrator 制定 Task Contract（静态，写入 contracts/TASK-ID.yaml）
2. pre-commit hook 自动执行 nexum-lint.sh 校验 Contract YAML
3. 注册任务到 active-tasks.json（status: pending，记录 contract_path）
4. dispatch generator（dispatch.sh）：
   a. 自动初始化（如 active-tasks.json 不存在）
   b. 记录 base_commit 到 active-tasks.json
   c. status → running，updated_at
   d. 根据 contract.type 选择 prompt 模板（coding / writing）
   e. 渲染 prompt，通过 tmux 向 agent session 发送命令
5. generator 执行，git commit（只 add scope.files）
6. post-commit hook → 写 nexum/events.jsonl + push
7. on-complete（generator 完成）：
   a. 校验 commit scope（diff base_commit..HEAD vs contract.scope.files）
      - 违规 → status: failed，reason: scope_violation，冻结现场，Telegram 通知人类
      - 正常 → 继续
   b. status → evaluating，updated_at
   c. dispatch evaluator（dispatch-evaluator.sh，根据 eval_strategy.type 选择 prompt 模板）
   d. 写 nexum/events.jsonl
   e. openclaw system event → 唤醒 orchestrator
8. evaluator 完成：
   a. 结果写入 nexum/runtime/eval/TASK-ID-iter-N.yaml
   b. on-complete 读取结果：
      - pass → status: done，解锁下游，通知，触发 harvest（批次全 done 时）
      - fail + iteration < max_iterations → iteration++，feedback 回 generator，status: running
        - retry 时根据 contract.type 重新路由 prompt 模板
      - fail + iteration >= max_iterations → status: escalated，Telegram 紧急通知
9. 所有任务 done → 汇报 → 自动触发 harvest（如 harvest.auto_trigger: true）
```

### 8.3 任务回滚（nexum-revert）

`scripts/nexum-revert.sh` 用于回滚失败或升级的任务：

```bash
nexum-revert.sh <task_id> [--force]
```

**前置条件**：任务状态必须为 `failed` 或 `escalated`，且有 `base_commit` 记录。

**执行流程**：
1. 展示 `base_commit..HEAD` 的 commit 列表
2. 交互式确认（`--force` 跳过）
3. `git revert base_commit..HEAD --no-commit` + 单次 commit
4. `git push`
5. 自动升级下游 blocked 任务：遍历 `active-tasks.json`，将 `depends_on` 包含被回滚 task id 且状态为 `blocked` 的任务标记为 `escalated`
6. Telegram 通知
7. 被回滚任务状态 → `cancelled`

### 8.4 重启恢复

orchestrator 启动时执行 reconcile：
1. 读取 `active-tasks.json`，找出所有 `status: running` 或 `evaluating` 的任务
2. 检查对应 tmux session 是否存活
3. session 不存活 → status: failed，reason: session_dead，通知人类
4. session 存活 → 视为正常继续（heartbeat 机制会检测是否卡死）

## 十一、Git Hooks

`install-hooks.sh` 安装两个 git hook：

### 11.1 pre-commit hook

- 检测暂存区是否包含 `docs/nexum/contracts/*.yaml` 文件
- 对每个匹配文件执行 `nexum-lint.sh --contract <path>`
- 任一 Contract 校验失败 → 阻止提交（exit 1）
- 无 Contract 变更时静默通过

### 11.2 post-commit hook

- 将 commit 事件写入 `nexum/events.jsonl`（task_id、hash、timestamp）
- 执行 `git push origin main`（失败不阻塞，仅记录 push_failed 事件）
- 不执行任何 git commit 操作（避免递归触发）

## 十二、开发路线图

### Phase 1 — 合约基础（MVP）✅

- Task Contract YAML 规范（coding 类型）
- `nexum-init.sh` 命令（项目结构 + AGENTS.md + hooks）
- Contract-based dispatch（含 base_commit 记录、scope.files 限定 commit）
- AGENTS.md/CLAUDE.md 同步（dispatch.sh force-commit 阶段）
- Evaluator agent（`review` + `unit` 模式，Codex 驱动）
- Eval result 持久化到 `nexum/runtime/eval/`
- 事件日志持久化到 `nexum/events.jsonl`
- Lesson 写入协议（模板 + AGENTS.md 内置学习指令）
- 重启恢复（reconcile 逻辑）
- 项目级 config.json 覆盖机制
- `nexum config --show` 命令

### Phase 2 — 验证能力扩展 ✅

- `eval_strategy.type: e2e` — Playwright 端到端评估（需 `local_url`），专用 prompt 模板 `prompt-evaluator-e2e.md`
- `eval_strategy.type: integration` — 组件集成测试（`shell:`/`review:` method 前缀），专用 prompt 模板 `prompt-evaluator-integration.md`
- `eval_strategy.type: composite` — 多子策略顺序执行、合并 verdict，专用 prompt 模板 `prompt-evaluator-composite.md`；不允许嵌套 composite

### Phase 3 — 知识系统 ✅

- `scripts/harvest.sh` — Lesson 收割流水线：
  - 读取 `docs/lessons/` 全部 lesson 文件
  - 分类：技术层（`affected` 字段）vs 流程层（`agent_note` 字段）
  - TF-IDF cosine similarity 去重（阈值 0.65），纯标准库实现
  - 矛盾检测（trigger similarity ≥ 0.55 + 否定词方向相反）
  - 无冲突时自动更新 `AGENTS.md` nexum:lessons 区间
  - 有冲突时 Telegram 通知人工审核（DUPLICATE / CONTRADICTION）
  - 状态持久化：`nexum/.harvest-state.json`
- `references/prompt-gardener.md` — Gardener agent prompt 模板
- `scripts/dispatch-gardener.sh` — 调度 gardener 修剪共享 lesson 库
- `on-complete.sh` — 批次全部 done 时自动触发 harvest（如 `harvest.auto_trigger: true`）

### Phase 4 — 任务类型扩展 ✅

- `type: creative` 和 `type: task` Contract 类型支持
- `cc-writer` agent（claude-sonnet-4-6）用于写作/创作任务
- `references/prompt-generator-writing.md` — 写作类 generator prompt 模板
- `dispatch.sh` 根据 `contract.type` 路由 prompt 模板（coding → coding 模板，creative/task → writing 模板）
- `on-complete.sh` retry 同样按 `contract.type` 路由
- Evaluator 支持 `threshold: score`（1-10 评分制）：
  - `min_score` 未指定时默认为 6
  - 创作类 eval 可通过 `eval_strategy.evaluator_model` 配置使用 claude-sonnet

### Phase 5 — 工程加固 ✅

- `scripts/nexum-lint.sh` — Contract + 项目结构校验器：
  - 17 项 Contract 检查（id 格式、type、name、时间戳、scope、deliverables、eval_strategy、criteria、agent 匹配等）
  - 5 项项目结构检查（AGENTS.md、hooks、nexum 目录）
  - Exit 0 = 无 error；exit 1 = 有 error；warning 不影响退出码
  - 修复建议默认显示（`--no-fix-hints` 抑制）
- `install-hooks.sh` 安装 post-commit（events + push）AND pre-commit（nexum-lint 校验暂存的 Contract YAML）
- `harvest.sh` 去重从简单方案升级到 TF-IDF cosine（纯标准库，无 pip 依赖）
- `nexum-revert.sh` 增强：
  - `--force` 标志跳过交互确认
  - 接受 `failed` 和 `escalated` 两种状态
  - 自动升级下游 `blocked` 任务为 `escalated`
  - 回滚后 Telegram 通知
