## 一、设计概览

### 1.1 系统边界

```
┌─────────────────────────────────────────────────────────────────┐
│  OpenClaw（宿主）                                                │
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │  OpenNexum Skill                                           │ │
│  │                                                            │ │
│  │  orchestrator (AI)  ←→  scripts/  ←→  nexum/              │ │
│  │         ↕                                                  │ │
│  │  tmux sessions ──→ [codex-1] [cc-frontend] [eval] ...     │ │
│  └────────────────────────────────────────────────────────────┘ │
│                                                                  │
│  project repo (git)                                             │
│  ├── AGENTS.md / CLAUDE.md                                      │
│  ├── docs/lessons/  docs/nexum/contracts/                       │
│  └── nexum/ (active-tasks.json, events.jsonl, runtime/ ...)    │
└─────────────────────────────────────────────────────────────────┘
```

OpenNexum 作为 OpenClaw skill 存在，由三部分组成：
- **scripts/**：可执行脚本（dispatch、on-complete、harvest、lint、revert 等）
- **references/**：prompt 模板、schema 定义、skill 级默认值、跨项目 lessons
- **orchestrator**：AI（OpenClaw 主 session）按本文档的规则做决策

### 1.2 核心数据流

```
需求输入
  │
  ▼
[orchestrator] 制定 Contract → docs/nexum/contracts/TASK-ID.yaml（静态，创建后不变）
  │
  ▼
[dispatch.sh] auto-init → 记录 base_commit → CONTRACT_TYPE 路由模板 → 启动 generator
  │
  ▼
[tmux: generator] 执行 → git add scope.files → git commit
  │
  ▼
[pre-commit hook] nexum-lint 校验 contract YAML
[post-commit hook] 写 nexum/events.jsonl + push
  │
  ▼
[on-complete.sh] scope 校验 → dispatch evaluator → 写 events.jsonl → openclaw system event
  │
  ▼
[dispatch-evaluator.sh] eval_strategy.type 路由 → 选择 prompt 模板 → 注入上下文
  │
  ▼
[tmux: evaluator] 读 Contract + 执行 eval → 写 nexum/runtime/eval/TASK-ID-iter-N.yaml
  │
  ▼
[on-complete.sh] 读 eval 结果 → pass: mark done + 解锁下游
                              → fail: feedback loop / escalate
  │
  ▼（all done + harvest.auto_trigger: true）
[harvest.sh] TF-IDF 去重 + 矛盾检测 → 更新 AGENTS.md nexum:lessons 区间
  │
  ▼
[gardener] 知识库维护（定期清理过时/冗余条目）
```

## 三、核心数据结构

### 3.1 Task Contract（YAML，静态）

Contract 创建后**不再修改**。运行态字段（status、iteration、commit 等）存储在 active-tasks.json。

```yaml
# docs/nexum/contracts/TASK-001.yaml
id: TASK-001
name: "实现用户登录页面"
type: coding                          # coding | task | creative
created_at: "2026-03-27T09:00:00Z"

scope:
  files:
    - src/pages/login.tsx
    - src/api/auth.ts
  boundaries:
    - "不修改 src/components/ui/ 下的基础组件"
  conflicts_with: []            # 与哪些 task id 存在语义冲突，需串行

deliverables:
  - "登录表单，支持邮箱 + 密码"
  - "调用 POST /api/auth/login"
  - "错误状态展示"
  - "登录成功跳转 /dashboard"

eval_strategy:
  type: e2e                     # unit | integration | e2e | review | composite
  tools: [playwright]
  local_url: "http://localhost:3000"
  criteria:
    - id: C1
      desc: "表单提交后正确调用 /api/auth/login"
      method: "playwright: intercept network request"
      threshold: pass
    - id: C2
      desc: "错误密码时展示错误提示"
      method: "playwright: check error element visible"
      threshold: pass
  sub_strategies: []            # composite 模式下的子策略列表

generator: codex-frontend       # coding → codex-*; creative/task → cc-writer
evaluator: eval
max_iterations: 3
depends_on: []
```

**Contract 类型与 agent 约束**：
- `coding` → generator 须为非 cc-writer agent（如 codex-frontend、codex-1 等）
- `creative` / `task` → generator 须为 cc-writer
- 类型不匹配时 nexum-lint 报 `TYPE_AGENT_MISMATCH` 错误

### 3.2 Active Tasks（JSON，唯一运行态主存储）

```json
{

  "project": "PolyGo",
  "repo": "git@github.com:ayaoplus/PolyGo.git",
  "created_at": "2026-03-27T09:00:00Z",
  "tasks": [

      "tmux_session": "nexum-codex-frontend",
      "eval_tmux_session": "nexum-eval",
      "eval_result_path": "nexum/runtime/eval/TASK-001-iter-2.yaml",
      "started_at": "2026-03-27T09:00:00Z",
      "completed_at": "2026-03-27T09:45:00Z",
      "updated_at": "2026-03-27T09:45:00Z",
      "last_error": null
    }
  ]
}
```

> **注意**：`agent-pool.json` 已废弃。当前哪个 agent 在跑什么 task，从 `active-tasks.json` 中 `status: running` + `tmux_session` 字段反查即可。

**完整 status 枚举**：`pending | blocked | running | evaluating | done | failed | escalated | cancelled`

**可更新字段**：`base_commit`, `commit_hash`, `tmux_session`, `eval_tmux_session`, `eval_result_path`, `started_at`, `completed_at`, `last_error`, `iteration`

### 3.3 Eval Result（YAML，持久化）

evaluator 写入 `nexum/runtime/eval/TASK-ID-iter-N.yaml`，路径由 on-complete.sh 按规则构造。

```yaml
# nexum/runtime/eval/TASK-001-iter-1.yaml
task_id: TASK-001
iteration: 1
evaluator_model: gpt-5.4
evaluated_at: "2026-03-27T09:30:00Z"

verdict: fail                   # pass | fail | error | inconclusive

# 系统层面的异常（app 未启动、工具不可用等）
system_errors: []

# 分层结果（composite 时有多个 strategy）
strategy_results:
  - strategy_type: e2e
    verdict: fail
    criteria_results:
      - id: C1
        result: pass
        evidence: "Network request to /api/auth/login observed"
      - id: C2
        result: fail
        evidence: "Element #login-error not visible after wrong password"
        detail: "src/pages/login.tsx:87 — setError() called but CSS condition wrong"
        score: null

# 给 generator 的 feedback（fail 时必填）
feedback: |
  C2 失败：错误提示元素存在但未显示。
  位置：src/pages/login.tsx:87，setError() 被调用但 CSS 条件判断有误。
  期望：API 返回 401 时 #login-error 应变为 visible。

# 截图路径（e2e 时）
artifacts:
  - type: screenshot
    path: "nexum/runtime/screenshots/TASK-001-C2-fail.png"
```

## 四、核心脚本设计

### 4.0 Auto-Init（自动初始化）

dispatch.sh 和 dispatch-evaluator.sh 在启动时均执行 auto-init 检查：

```
1. 检查 nexum/active-tasks.json 是否存在
2. 不存在 → 自动调用 nexum-init.sh --project <name> --type coding
3. 已存在 → 跳过，正常继续
```

此机制确保首次 dispatch 时无需手动执行 init，降低操作门槛。

### 4.1 dispatch.sh

**职责**：派发 generator，管理任务状态转换，根据 Contract 类型选择 prompt 模板。

```bash
dispatch.sh <agent> <task_id> --role <generator|evaluator> \
  [--prompt-file <path>] <cli_command...>
```

执行流程：

```
1. Auto-init 检查（见 4.0）
2. 验证 task_id 格式（whitelist regex：^[A-Z]+-[0-9]+$）
3. 读取 Contract YAML，缓存为 JSON（CONTRACT_JSON_FILE = /tmp/nexum-contract-<task_id>.XXXXXX）
4. 根据 CONTRACT_TYPE 选择 prompt 模板：
   - creative | task → references/prompt-generator-writing.md
   - *（coding 或其他） → references/prompt-generator-coding.md
   - writing 模板不存在时 fallback 到 coding 模板（stderr 警告）
5. 检查 tmux session（nexum-<agent>）存在；不存在则创建
6. 检查 agent 是否 busy（active-tasks.json 中已有 status:running 且 tmux_session==nexum-<agent>）
   - busy → 拒绝，报告冲突
7. 原子更新 active-tasks.json（flock + tmpfile + rename）：
   - status → running，base_commit，tmux_session，started_at，updated_at
8. 写 nexum/events.jsonl（task_started）
9. 构建 prompt（详见 4.1.1）
10. 在 tmux session 中执行 CLI（heredoc，不做 shell 插值）
11. generator 完成后执行 force-commit（仅 add scope.files）
12. 调用 on-complete.sh（cleanup trap 兜底）
```

**4.1.1 Prompt 构建（render_prompt）**

```
读取 prompt 模板（按 CONTRACT_TYPE 路由选择）
  + 注入 {{TASK_NAME}}：contract.name
  + 注入 {{SCOPE_FILES}}：scope.files 列表
  + 注入 {{SCOPE_BOUNDARIES}}：scope.boundaries 列表
  + 注入 {{DELIVERABLES}}：deliverables 列表
  + 注入 {{CRITERIA_PREVIEW}}：eval_strategy.criteria 预览
  + 注入 {{RELEVANT_LESSONS}}：相关 lessons（从 docs/lessons/ 筛选）
  + 注入 {{GIT_COMMIT_CMD}}：预构建的 git commit 命令（仅 scope.files）
  + 注入 {{EVAL_RESULT_PATH}}：eval 结果写入路径
写入 /tmp/nexum-prompt-<task_id>.txt（可丢弃）
```

**4.1.2 Force-commit 兜底**

```bash
# 仅 add Contract 中 scope.files 列出的文件
if [ -n "$(git status --porcelain)" ]; then
  git add -- ${SCOPE_FILES[@]}
  git commit -m "chore(nexum): force-commit for $TASK_ID [skip ci]"
fi

# 同步 CLAUDE.md
if git diff HEAD~1 --name-only 2>/dev/null | grep -q "^AGENTS.md$"; then
  cp AGENTS.md CLAUDE.md
  git add CLAUDE.md
  git commit -m "chore(nexum): sync CLAUDE.md from AGENTS.md" --no-verify
fi
```

**注意**：CLAUDE.md 同步在 dispatch.sh 显式处理，**不在 post-commit hook 中执行**，避免递归 commit。

### 4.2 on-complete.sh

**职责**：generator 或 evaluator 完成后的统一回调。

```bash
on-complete.sh <task_id> <exit_code> --role <generator|evaluator>
```

**场景 A：generator 完成**

```
1. exit_code != 0 → status: failed，last_error，写 events.jsonl，通知人类，退出
2. Scope 校验：
   git diff <base_commit>..HEAD --name-only 与 contract.scope.files 对比
   - 有 scope 外文件 → status: failed，reason: scope_violation
     冻结现场（不 revert），写 events.jsonl
     Telegram 通知：⚠️ TASK-001 scope 违规，冻结，附违规文件列表
     提示：执行 nexum revert TASK-001 可回滚（人类决定是否执行）
     退出
3. 更新 active-tasks.json：status → evaluating，commit_hash，updated_at
4. 写 nexum/events.jsonl（eval_started）
5. dispatch evaluator（dispatch-evaluator.sh）
6. openclaw system event → 唤醒 orchestrator
7. Telegram 通知（compact：generator done，正在 eval）
```

**场景 B：evaluator 完成**

```
1. 按约定路径读取 eval result：nexum/runtime/eval/TASK-ID-iter-N.yaml
   - 文件不存在 → status: failed，reason: eval_result_missing，通知人类
2. 解析 verdict：
   pass：
     a. status: done，completed_at，updated_at
     b. 写 nexum/events.jsonl（task_done）
     c. 扫描 active-tasks.json，所有 blocked 任务中 depends_on 全 done 的 → pending
     d. openclaw system event
     e. Telegram 完成通知（含 iteration 数、耗时、token）
     f. 检查是否批次全 done → 触发 harvest（见 4.2.1）

   fail + iteration < max_iterations：
     a. iteration++，updated_at
     b. 构建 feedback prompt（含 eval_result.feedback 全文）
     c. 重试模板选择：creative/task 类型使用 writing 模板，coding 类型使用 coding 模板
     d. dispatch generator（重试），status: running
     e. 写 nexum/events.jsonl

   fail + iteration >= max_iterations：
     a. status: escalated，updated_at
     b. 写 nexum/events.jsonl
     c. openclaw system event
     d. Telegram 紧急通知（含完整 eval feedback + 操作建议）

   error / inconclusive：
     a. 视为 fail，同 fail 分支处理
     b. 通知中附加 system_errors 内容
```

**4.2.1 批次完成与 Harvest 触发**

```
1. update-task-status.sh 以 --output-batch-done 调用
2. 若输出包含 BATCH_DONE=true：
   a. 读取 config 中 harvest.auto_trigger（默认 true）
   b. auto_trigger == true → 调用 harvest.sh
   c. auto_trigger == false → 仅 Telegram 通知批次完成
```

### 4.3 dispatch-evaluator.sh

**职责**：根据 Contract eval_strategy 选择 evaluator prompt 模板并派发。

```bash
dispatch-evaluator.sh <task_id>
```

执行流程：
```
1. Auto-init 检查（见 4.0）
2. 读取 Contract，获取 eval_strategy.type
3. 构造 eval_result_path：nexum/runtime/eval/TASK-ID-iter-N.yaml（N 从 active-tasks 读）
4. 选择 prompt 模板并注入特定变量：
   - e2e → references/prompt-evaluator-e2e.md
     注入 {{LOCAL_URL}}（contract.eval_strategy.local_url，fallback: config eval.default_local_url）
     注入 {{TASK_ID}}（用于截图路径构造）
   - integration → references/prompt-evaluator-integration.md
   - composite → references/prompt-evaluator-composite.md
     注入 {{SUB_STRATEGIES_LIST}}（子策略类型列表）
     注入 {{SUB_STRATEGIES_CRITERIA}}（各子策略的 criteria）
   - *（unit/review 等） → references/prompt-evaluator.md
5. 通用注入：
   - contract.deliverables
   - contract.eval_strategy.criteria（逐条，含 method 和 threshold）
   - eval_result_path（告知 evaluator 结果写到哪里）
6. 调用 dispatch.sh <evaluator-agent> <task_id> --role evaluator <cli_command>
```

### 4.4 update-task-status.sh

所有对 `active-tasks.json` 的写操作通过此脚本，保证原子性：

```bash
update-task-status.sh <task_id> <status> [key=value ...] [--output-batch-done]
```

内部实现：
```bash
LOCK_FILE="$TASK_FILE.lock"
(
  flock -x 200
  python3 -c "
import json, sys, os
with open('$TASK_FILE') as f:
    data = json.load(f)
for t in data['tasks']:
    if t['id'] == '$TASK_ID':
        t['status'] = '$STATUS'
        t['updated_at'] = '$(date -u +%Y-%m-%dT%H:%M:%SZ)'
        # 额外 key=value 字段...
        break
tmp = '$TASK_FILE.tmp'
with open(tmp, 'w') as f:
    json.dump(data, f, indent=2, ensure_ascii=False)
os.replace(tmp, '$TASK_FILE')
"
) 200>"$LOCK_FILE"
```

- task 不存在时返回 exit 2
- `--output-batch-done`：当所有任务均为终态时，stdout 输出 `BATCH_DONE=true`

### 4.5 Git Hooks（install-hooks.sh）

install-hooks.sh 安装两个 hook：

**pre-commit hook（Phase 5）**

对 staging area 中的 contract YAML 文件执行 nexum-lint 校验：

```bash
# 检查 staged 中匹配 docs/nexum/contracts/*.yaml 的文件
# 逐一调用 nexum-lint.sh 校验
# 任一校验失败 → exit 1，阻止 commit
```

**post-commit hook**

hook 中**不做任何 commit 操作**，只做两件事：

```bash
#!/bin/bash
# 1. 写事件信号
TASK_ID="${NEXUM_TASK_ID:-unknown}"
echo "{\"event\":\"commit\",\"task_id\":\"$TASK_ID\",\"hash\":\"$(git rev-parse HEAD)\",\"ts\":\"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"}" \
  >> nexum/events.jsonl

# 2. 推送（失败只记录，不阻塞）
git push origin main 2>/dev/null || \
  echo "{\"event\":\"push_failed\",\"task_id\":\"$TASK_ID\",\"ts\":\"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"}" \
  >> nexum/events.jsonl
```


### 4.8 nexum-revert.sh

回滚某个 task 的所有提交，并级联处理下游依赖：

```bash
nexum-revert.sh <task_id> [--force]
```

执行：
```
1. 读 active-tasks.json，获取 task.base_commit
2. 确认 task.status 为 failed 或 escalated（仅允许这两种状态回滚）
3. git log --oneline <base_commit>..HEAD（展示将被回滚的 commits）
4. 询问确认（交互模式）；--force 跳过确认
5. git revert <base_commit>..HEAD --no-commit
6. git commit -m "revert: roll back TASK-ID commits"
7. 更新 active-tasks.json：status: cancelled
8. 级联处理下游：
   - 扫描 active-tasks.json 中 status: blocked 且 depends_on 包含被回滚 task_id 的任务
   - 调用 update-task-status.sh <downstream_id> escalated last_error=upstream_<task_id>_cancelled
9. Telegram 通知完成
```

```
[注入 contract.deliverables]
[注入 contract.scope.files]

## Criteria to Verify
[逐条注入 criteria，含 id、desc、method、threshold]

## Evaluation Rules
1. 逐条验证，不允许跳过任何一条
2. pass 需要：具体证据（文件名/行号/输出内容）
3. fail 需要：文件路径 + 行号（如能确定）+ 期望 vs 实际

[注入 eval-result.yaml schema]
```

### 6.2 Harvest 数据流（Phase 3/5）

harvest.sh 是知识收割的核心脚本，将 agent 产出的 lessons 分类、去重、写入 AGENTS.md。

**Lesson 解析**

```
1. 扫描 docs/lessons/*.md（跳过 TEMPLATE.md）
2. 对比 .harvest-state.json 中 processed_files，仅处理增量
3. 通过 yaml-to-json.sh 将 bullet 格式字段转换为结构化 YAML
```

**分类规则**

| 条件 | 分类 |
|------|------|
| 有 `affected` 字段，无 `agent_note` | technical（技术层） |
| 有 `agent_note` 字段，无 `affected` | process（流程层） |
| 两者都有或都无 | invalid（跳过） |

**TF-IDF 去重检测**

```python
# 构建 IDF 语料库：所有已有技术 lesson 的 right + trigger 文本
# 对候选 lesson 计算 cosine similarity
if similarity(candidate["right"], existing["right"]) > 0.65:
    return "DUPLICATE"
```

**矛盾检测**

```python
# trigger 相似度 ≥ 0.55 且 right 字段存在否定词翻转
if (similarity(candidate["trigger"], existing["trigger"]) >= 0.55 and
    has_negation_flip(candidate["right"], existing["right"])):
    return "CONTRADICTION"
```

**AGENTS.md 更新**

```
1. 定位 <!-- nexum:lessons:start --> ... <!-- nexum:lessons:end --> 区间
2. git blame 检查：若区间内有上次 harvest 之后的人工修改 → 停止自动合并，通知人类
3. 原子写入：tmpfile + os.replace
4. git add AGENTS.md && git commit --no-verify
```

**冲突分流**

- `DUPLICATE` → Telegram 通知人工选择保留哪条
- `CONTRADICTION` → Telegram 通知人工仲裁
- 无冲突 → 自动写入 `nexum:lessons` 区间

**流程层 Lesson 镜像**

process 类型的 lesson 镜像到 `references/lessons/`，实现跨项目知识积累。

## 七、重启恢复设计

### 7.1 Reconcile 逻辑

orchestrator 启动（或手动触发 `nexum reconcile`）时执行：

```
1. 读取 active-tasks.json，找所有非终态任务（非 done/cancelled）
2. 对每个 running/evaluating 任务：
   a. 检查 tmux_session 是否存活（tmux has-session -t nexum-xxx）
   b. 存活 + updated_at < 15分钟前 → 视为正常，heartbeat 继续监控
   c. 存活 + updated_at > 15分钟前 → 疑似卡死，通知人类
   d. 不存活 → status: failed，reason: session_dead，通知人类
3. 读取 nexum/events.jsonl，找上次已记录的最后事件
   - 补发可能丢失的 openclaw system event
4. 对 blocked 任务：检查 depends_on 是否都 done，是则转 pending
5. 对 pending 任务：通知 orchestrator 可以 dispatch
```

### 7.2 数据恢复优先级

重启后的状态以 **active-tasks.json 为准**（因为它通过 flock+atomic-write 维护，是最可靠的）。events.jsonl 用于补充事件流，不作为状态的 source of truth。
