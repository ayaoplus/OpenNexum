## 一、设计概览

### 1.1 系统边界

```
┌─────────────────────────────────────────────────────────────────┐
│  OpenClaw（宿主）                                                │
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │  OpenNexum Skill                                           │ │
│  │                                                            │ │
│  │  orchestrator (AI)  ←→  scripts/  ←→  nexum/state/        │ │
│  │         ↕                                                  │ │
│  │  tmux sessions ──→ [codex-1] [cc-frontend] [eval] ...     │ │
│  └────────────────────────────────────────────────────────────┘ │
│                                                                  │
│  project repo (git)                                             │
│  ├── AGENTS.md / CLAUDE.md                                      │
│  ├── docs/lessons/ docs/nexum/contracts/                        │
│  └── nexum/ (active-tasks.json, config.json ...)               │
└─────────────────────────────────────────────────────────────────┘
```

OpenNexum 作为 OpenClaw skill 存在，主要由三部分组成：
- **scripts/**：可执行脚本（dispatch、on-complete、harvest、init 等）
- **references/**：prompt 模板、schema 定义、lesson 模板
- **orchestrator**：AI（OpenClaw 主 session）按本文档的规则做决策

### 1.2 核心数据流

```
需求输入
  │
  ▼
[orchestrator] 制定 Contract → contracts/TASK-ID.yaml
  │
  ▼
[dispatch.sh] 注册 + 启动 generator
  │
  ▼
[tmux: generator] 执行 → git commit
  │
  ▼
[post-commit hook] push + 写 /tmp/nexum-signals.jsonl
  │
  ▼
[on-complete.sh] 更新状态 → dispatch evaluator → openclaw system event
  │
  ▼
[tmux: evaluator] 读 Contract → 逐条验证 → 写 eval-result.yaml
  │
  ▼
[on-complete.sh] 读 eval 结果 → pass/fail 分流 → openclaw system event
  │
  ▼
[orchestrator] 处理结果：mark done / feedback loop / escalate
  │
  ▼（all done）
[harvest.sh] + [gardener] 知识沉淀
```

## 三、核心数据结构

### 3.1 Task Contract（YAML）

```yaml
# docs/nexum/contracts/TASK-001.yaml

# ── 元数据 ──────────────────────────────────────
id: TASK-001                    # 格式：[A-Z]+-[0-9]+，不允许空格
name: "实现用户登录页面"
type: coding                    # coding | writing | analysis | creative
created_at: "2026-03-27T09:00:00Z"
status: pending                 # pending | running | evaluating | done | failed | escalated

# ── 执行配置 ─────────────────────────────────────
generator: codex-frontend       # agent 名称，对应 config.json agents.*
evaluator: eval
max_iterations: 3               # evaluator fail 后最多重试次数
depends_on: []                  # 依赖的 task id 列表

# ── 范围定义 ─────────────────────────────────────
scope:
  files:                        # generator 可修改的文件列表
    - src/pages/login.tsx
    - src/api/auth.ts
  boundaries:                   # 明确禁止修改的范围
    - "不修改 src/components/ui/ 下的基础组件"
    - "不修改数据库 schema"

# ── 交付物 ───────────────────────────────────────
deliverables:
  - "登录表单，支持邮箱 + 密码"
  - "调用 POST /api/auth/login"
  - "错误状态展示"
  - "登录成功跳转 /dashboard"

# ── 验证策略 ─────────────────────────────────────
eval_strategy:
  type: e2e                     # unit | integration | e2e | review | composite
  tools: [playwright]           # 验证工具
  local_url: "http://localhost:3000"  # e2e 必填
  criteria:
    - id: C1
      desc: "表单提交后正确调用 /api/auth/login"
      method: "playwright: intercept network request"
      threshold: pass           # pass | score（写作类用 score）
    - id: C2
      desc: "错误密码时展示错误提示"
      method: "playwright: check error element visible"
      threshold: pass
    - id: C3
      desc: "登录成功后 URL 变为 /dashboard"
      method: "playwright: check URL after submit"
      threshold: pass

# ── 运行时字段（系统自动填写）────────────────────
iteration: 0                    # 当前是第几次迭代
started_at: ~
completed_at: ~
tmux_session: ~                 # generator 使用的 tmux session
eval_tmux_session: ~            # evaluator 使用的 tmux session
commit_hash: ~                  # generator 最后一次 commit
eval_result_path: ~             # evaluator 输出文件路径
```

## 四、核心脚本设计

### 4.1 dispatch.sh

**职责**：派发 generator 或 evaluator，管理任务状态转换。

```
dispatch.sh <agent> <task_id> [--contract <path>] [--eval] <cli_command...>
```

执行流程：
```
1. 验证 task_id 格式（whitelist regex：^[A-Z]+-[0-9]+$）
2. 读取 Contract YAML，验证 agent 与 contract.generator 匹配
3. 检查 tmux session 存在（nexum-<agent>）；不存在则创建
4. 原子更新 active-tasks.json：status → running，tmux_session，started_at
5. 更新 agent-pool.json：agent status → busy，current_task
6. 构建 prompt：
   a. 读取 prompt 模板（references/prompt-generator-*.md）
   b. 注入 Contract 内容（scope, deliverables, criteria）
   c. 注入项目上下文（AGENTS.md 摘要 + 相关 lessons）
   d. 写入 /tmp/nexum-prompt-<task_id>.txt
7. 在 tmux session 中执行 CLI（通过 heredoc，避免 shell 注入）
8. 执行后追加 force-commit 检查（防 agent 忘记 commit）
9. 调用 on-complete.sh（cleanup trap 确保即使 exit != 0 也执行）
```

**Prompt 构建细节**：
- Contract 内容以结构化方式注入，不是自由文本
- 从 `docs/lessons/` 中筛选 tags 与当前任务相关的 lessons 注入
- 注入完整的 `scope.files` 和 `scope.boundaries`，明确写 "DO NOT modify" 列表
- 预写 commit message 格式，不让 agent 自由发挥

### 4.2 on-complete.sh

**职责**：任务完成后的统一回调，负责状态更新、evaluator 派发、通知。

```
on-complete.sh <task_id> <exit_code> [--eval-result <path>]
```

执行流程：

**场景 A：generator 完成**
```
1. 解析 exit_code
   - exit 0 → 继续
   - exit != 0 → status → failed，通知人类，退出
2. 验证 git commit scope（git diff HEAD~1 --stat 与 contract.scope.files 对比）
   - scope 违规 → git revert HEAD，status → failed，通知人类
3. 更新状态：status → evaluating，commit_hash
4. 更新 agent-pool.json：generator status → idle
5. 调用 dispatch-evaluator.sh <task_id>
6. openclaw system event --text "Generator done: <task_id>" --mode now
7. Telegram 通知（compact 格式）
```

**场景 B：evaluator 完成**
```
1. 读取 eval-result.yaml（从 contract.eval_result_path）
2. 解析 verdict
   - pass：
     a. status → done，completed_at
     b. 更新 agent-pool.json：evaluator status → idle
     c. 自动解锁 depends_on 包含此 task_id 的任务（blocked → pending）
     d. openclaw system event
     e. Telegram 通知（含 token 用量 + 耗时）
     f. 触发 harvest（如果所有任务 done）
   - fail：
     a. contract.iteration += 1
     b. iteration <= max_iterations：
        - 构建 feedback prompt（含 eval_result.feedback）
        - dispatch generator（重试）
        - status → running
     c. iteration > max_iterations：
        - status → escalated
        - Telegram 紧急通知（含完整 eval 反馈）
        - openclaw system event
```

### 4.3 dispatch-evaluator.sh

**职责**：根据 Contract 的 eval_strategy 决定如何派发 evaluator。

```
dispatch-evaluator.sh <task_id>
```

执行流程：
```
1. 读取 Contract，获取 eval_strategy.type
2. 选择 evaluator prompt 模板：
   - unit/integration/review → references/prompt-evaluator.md
   - e2e → references/prompt-evaluator-e2e.md（含 Playwright 操作指引）
   - composite → 按 sub-strategies 逐个派发，汇总结果
3. 注入 Contract criteria（逐条，含 method 和 threshold）
4. e2e 模式：额外注入 local_url，要求 evaluator 验证 app 已启动
5. dispatch.sh eval <task_id> --eval <cli_command>
```

### 4.4 harvest.sh

**职责**：收割新 lessons，去重，更新 AGENTS.md。

```
harvest.sh [--project <path>] [--dry-run]
```

执行流程：
```
1. 扫描 docs/lessons/，找出上次 harvest 后新增的文件
   （通过 nexum/harvest-state.json 记录 last_harvest_at）
2. 解析每条 lesson，按字段分流：
   - 有 affected → 技术层
   - 有 agent_note → 流程层
3. 技术层处理：
   a. 与 docs/lessons/index.md 做关键词 + tags 去重
   b. 与 AGENTS.md 现有内容做互斥检查
      - 检测冲突：新规则与已有规则描述方向相反
      - 检测重复：相似度 > 0.8（简单 trigram 匹配）
   c. 无冲突 + 无重复 → 追加到 AGENTS.md 对应 section
      同时触发 CLAUDE.md 同步（cp AGENTS.md CLAUDE.md）
   d. 有冲突 → 生成 diff 文件，Telegram 通知待审核
4. 流程层处理：
   a. 追加到 skill/references/lessons/
   b. 更新 SKILL.md 相关章节（如 dispatch 注意事项）
5. 更新 nexum/harvest-state.json（last_harvest_at, processed_count）
6. Telegram 汇报：N 条 lesson 已自动合并，M 条待审核
```

### 4.5 nexum-init.sh

**职责**：初始化项目的 OpenNexum 结构。

```
nexum-init.sh --project <name> --tech "<stack>" --type <coding|writing|mixed>
```

执行流程：
```
1. 创建目录结构（docs/design, docs/lessons, docs/nexum/contracts, nexum/history）
2. 生成 AGENTS.md：
   a. 从 references/agents-md-template.md 读取基础模板
   b. 从 skill/references/lessons/ 筛选与 tech stack 相关的 lessons
   c. 写入项目名、tech stack、日期
   d. 写入错误学习协议
3. 生成 CLAUDE.md（cp AGENTS.md CLAUDE.md）
4. 生成 nexum/config.json（从默认值填充）
5. 生成 docs/lessons/TEMPLATE.md（lesson 写入模板）
6. 初始化 nexum/active-tasks.json（空批次）
7. 初始化 nexum/agent-pool.json（所有 agent idle）
8. 安装 git hooks（install-hooks.sh）
9. 创建初始 tmux sessions
10. Telegram 通知：项目已初始化
```

### 4.6 install-hooks.sh

安装两个 git hook：

**post-commit**：
```bash
#!/bin/bash
SKILL_DIR=~/.openclaw/workspace/skills/opennexum

# 1. 自动同步 CLAUDE.md
if git diff HEAD~1 --name-only | grep -q "^AGENTS.md$"; then
  cp AGENTS.md CLAUDE.md
  git add CLAUDE.md
  git commit --amend --no-edit --no-verify
fi

# 2. 自动 push
git push origin main

# 3. 写信号
echo "{\"event\":\"commit\",\"task_id\":\"${NEXUM_TASK_ID}\",\"hash\":\"$(git rev-parse HEAD)\",\"ts\":\"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"}" \
  >> /tmp/nexum-signals.jsonl
```

**pre-commit**（可选，轻量门禁）：
```bash
#!/bin/bash
# TypeScript 项目：tsc 类型检查
# 检查失败时阻止 commit，提示 agent 修复
```

## 五、Evaluator 设计

### 5.1 Evaluator Prompt 结构

```
## Role
你是一个独立代码/内容评审员。你的工作是找问题，不是找夸奖的理由。
对于每条 criterion，你必须给出明确的 pass 或 fail，以及具体的证据。
"整体看起来没问题"不是有效结论。

## Task Contract
[注入 contract.deliverables]
[注入 contract.scope.files]

## Criteria to Verify
[逐条注入 criteria，含 id、desc、method]

## Evaluation Rules
1. 逐条验证，不允许跳过
2. pass 需要：有具体证据支持
3. fail 需要：文件路径 + 行号（如能确定）+ 期望行为 vs 实际行为
4. 所有 criteria pass → verdict: pass
5. 任意 criterion fail → verdict: fail
6. 输出格式必须是 YAML，写入 [eval_result_path]

## Output Format
[注入 eval-result.yaml schema]
```

### 5.2 E2E Evaluator 扩展

e2e 模式下，evaluator 额外获得 Playwright MCP 能力：

```
## E2E Testing Protocol
App URL: [local_url]

在开始评估前：
1. 确认 app 已启动（访问 local_url，检查响应）
2. 如 app 未启动，在 eval result 中记录 "app_not_running: true"，verdict: fail

对于每条 e2e criterion：
1. 按照 method 描述执行 Playwright 操作
2. 截图保存到 /tmp/nexum-eval-screenshots/TASK-ID-C1.png
3. 记录实际观察到的行为
```

### 5.3 Writing/Creative Evaluator 评分标准

写作类任务使用评分制（threshold: score）：

```yaml
criteria:
  - id: C1
    desc: "内容完整性"
    method: "review: check all deliverables addressed"
    threshold: score
    min_score: 7    # 满分 10
  - id: C2
    desc: "表达清晰度"
    method: "review: assess clarity and structure"
    threshold: score
    min_score: 7
  - id: C3
    desc: "原创性（非模板化输出）"
    method: "review: check for generic AI patterns"
    threshold: score
    min_score: 6
```

evaluator 对每条给出 1-10 分 + 说明；低于 min_score → fail。

## 项目概览

### 6.3 Harvest 冲突检测逻辑

```python
# 伪代码
def check_conflict(new_lesson, existing_lessons):
    # 1. 关键词重叠检测
    new_keywords = extract_keywords(new_lesson.right)
    for existing in existing_lessons:
        overlap = jaccard_similarity(new_keywords, extract_keywords(existing.right))
        if overlap > 0.8:
            return ConflictType.DUPLICATE, existing

    # 2. 互斥检测（简单否定词匹配）
    # 如果 new.right 和 existing.right 描述方向相反
    if contradicts(new_lesson.right, existing.right, same_trigger=True):
        return ConflictType.CONTRADICTION, existing

    return None
```

初期实现用关键词匹配（不依赖 embedding），后期 Phase 5 升级为 embedding-based。

## 八、并行安全

### 8.1 文件锁

所有对 `nexum/active-tasks.json` 和 `nexum/agent-pool.json` 的写操作通过 `flock` + tmpfile + rename 原子执行：

```bash
LOCK_FILE="$TASK_FILE.lock"
(
  flock -x 200
  # 读 → 修改 → 写 tmpfile → mv tmpfile → 原子替换
  python3 -c "..." > "$TASK_FILE.tmp"
  mv "$TASK_FILE.tmp" "$TASK_FILE"
) 200>"$LOCK_FILE"
```

### 8.2 Scope 冲突检测

dispatch 前检查：

```bash
# 读取所有 running 任务的 contract.scope.files
# 与待 dispatch 任务的 contract.scope.files 求交集
# 有交集 → 拒绝并行，报告冲突文件
```

### 8.3 Agent 忙碌检测

dispatch 前检查 agent-pool.json：
- `status: busy` → 等待或选择备用 agent（codex-2 等扩展 agent）
- `status: dead` → health-check.sh 处理，通知人类

## 十二、错误处理

### 12.1 Agent 退出异常

dispatch.sh 通过 `trap` 捕获非正常退出：

```bash
cleanup() {
  local exit_code=$?
  if [ $exit_code -ne 0 ]; then
    update-task-status.sh "$TASK_ID" "failed"
    update-agent-status.sh "$AGENT" "idle"
    # 通知 orchestrator
    openclaw system event --text "TASK $TASK_ID failed (exit $exit_code)" --mode now
  fi
}
trap cleanup EXIT
```

### 12.2 Force-Commit 兜底

generator 完成后，dispatch.sh 检查 git status：
```bash
if [ -n "$(git status --porcelain)" ]; then
  # agent 忘记 commit，强制提交
  git add -A
  git commit -m "chore(nexum): force-commit by dispatch.sh for $TASK_ID"
fi
```

### 12.3 Contract 验证

dispatch.sh 在派发前验证 Contract YAML：
- `id` 格式合法
- `generator` 与调用 agent 一致
- `eval_strategy.type` 为已知值
- e2e 类型必须有 `local_url`
- `criteria` 不为空

验证失败 → 拒绝派发，通知 orchestrator。
